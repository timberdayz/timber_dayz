# TikTok 双页日历日期组件建模指南

## 目的

本文总结 TikTok `products` 数据域日期控件的真实建模经验，用于：

- 后续 TikTok 其他数据域日期组件开发
- 其他平台双页日历组件开发
- 组件调试时快速识别错误抽象和常见陷阱

## 结论先行

TikTok `products` 双页日历不能按“点数字”建模，必须按“点完整日期”建模。

错误抽象：

- `pane + day`
- `body + day`
- `selector + first visible`

正确抽象：

- `left anchor month`
- `left grid`
- `right grid`
- `active boundary`
- `exact target date`

## 本次确认的最小模型

对 TikTok `products` 双页日历，最小且正确的模型是：

1. 左页顶部月份是唯一月份锚点
2. 右页月份永远等于左页月份 `+1 月`
3. 同月范围：
   - 开始日期点左页
   - 结束日期点左页
4. 跨月范围：
   - 开始日期点左页
   - 结束日期点右页

注意：

- 不要把右页月份当作第二个独立判断源
- 右页月份应从左页月份推导

## 真实故障暴露出的根问题

在 TikTok `products` 真实调试过程中，出现过这些典型失败：

- 开始日期本应点左页，却点到了右页同数字日期
- 结束日期本应点右页，却点到了左页同数字日期
- 右页日期可见，但代码仍然报告“failed to select end day on right page”
- 成功点击后因为实现 bug 被误报为失败

这些问题的共同根因不是“少一个 selector”，而是抽象层级错误：

- 系统仍然把双页日历当成“容器 + 数字点击器”
- 没有把“左页锚点月份 + 左右 grid + active boundary + 完整日期”建立成唯一状态对象

## 正确的开发顺序

### 1. 先取证，再建模

优先看 `pwcli` 证据包，不要先猜 UI。

至少需要确认：

- 打开态是否是双页日历
- 左右页顶部月份显示方式
- 开始/结束输入框 active 状态
- 点击开始日期后，弹层结构是否变化

### 2. 建立唯一状态对象

推荐最少包含：

- `left_month`
- `right_month`
- `left_grid`
- `right_grid`
- `active_boundary`

这里的 `right_month` 可以由 `left_month + 1` 推导，但状态对象中仍可存下这个派生结果，供断言使用。

### 3. 先验证不变量，再点击

点击前必须先验证：

- `right_month == next(left_month)`
- 当前 `active_boundary` 正确
- 左右 grid 都存在

如果这些前提不成立，不要直接点。

### 4. 开始和结束是两个独立阶段

不要共享同一份“点击前识别结果”。

正确流程：

1. 打开弹层
2. 聚焦开始输入框
3. 校验 `active_boundary == start`
4. 导航左页到开始月份
5. 重新读取状态
6. 在左 grid 中点击开始日期
7. 验证 `start_input == target_start`
8. 聚焦结束输入框
9. 校验 `active_boundary == end`
10. 重新读取状态
11. 同月则点左 grid，跨月则点右 grid
12. 验证 `end_input == target_end`

## 为什么“点数字”会失败

双页日历中，左右页会同时出现相同数字。

例如跨月场景：

- 左页会有 `2026-03-02`
- 右页会有 `2026-04-02`

如果系统只知道“点数字 2”，那么无论你加多少 bbox / selector 兜底，最终都可能点错。

因此必须把 grid 中每个 cell 解释成完整日期：

- 左 grid 的 `2` 可能是 `2026-03-02`
- 右 grid 的 `2` 可能是 `2026-04-02`

点击对象必须是“完整日期对应的 cell”，而不是“数字相同的 cell”。

## 组件设计建议

### 推荐保留的能力

- 左右 panel 识别
- 左右 body/grid 识别
- active boundary 检查
- 顶部输入框值验证
- 点击后重新读取状态

### 不建议作为主流程依赖的能力

- page-global body 猜测
- 宽泛 selector 兜底
- “第一个可见数字”
- 仅按 bbox 猜是左还是右

这些能力可以保留为调试辅助，但不要作为主流程的核心逻辑。

## 对 Shopee 的正确参考方式

Shopee 值得借鉴的不是具体 selector，而是建模原则：

- 先锁定唯一 popup
- 再读当前 header / 当前状态
- 再导航
- 再点击
- 点击后立即验证

不要直接照搬 Shopee 的“导航到月后按数字点击”逻辑，因为 Shopee 的典型日历更接近单面板，而 TikTok `products` 是双页联动日历。

## 这次我们做对了什么

- 使用了真实 `pwcli` 证据包，而不是凭感觉猜 UI
- 把问题从“点击失败”逐步收敛到“建模对象错误”
- 把规则收敛成最小模型，而不是引入越来越多兜底
- 开始把实现从 selector 驱动改成状态驱动
- 把回归测试补到了真实失败场景上
- 修掉了一个会吞掉成功点击结果、导致误报失败的实现 bug

## 后续开发检查清单

开发任何双页日历组件前，至少检查以下几点：

- 是否存在双页月份展示
- 左页是否可以作为唯一月份锚点
- 右页月份是否可由左页推导
- 开始日期是否固定左页
- 结束日期是否按同月/跨月固定左或右
- 点击后是否重新读取状态
- 顶部输入框是否作为最终真相源
- 是否已经把 cell 映射成完整日期，而不是仅按数字点击

## 一句话版本

双页日历的根本不是“找左边和右边的数字”，而是：

**先建立左页月份锚点，再在正确的 grid 里点击完整日期。**
