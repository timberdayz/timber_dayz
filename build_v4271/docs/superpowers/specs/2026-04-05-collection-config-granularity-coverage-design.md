# Collection Config Granularity And Coverage Design

**日期:** 2026-04-05

**目标:** 将采集配置从“单一总表 + 手动配置”升级为“按粒度管理 + 按店铺账号配置 + 按覆盖率巡检”的结构，同时保留当前手动配置与有头测试配置的灵活性。

## 1. 背景

当前采集配置已经具备以下能力：

- 可创建采集配置
- 可选择店铺账号执行任务
- 可手动指定数据域、时间范围、执行模式
- 可作为测试和临时采集入口

但在店铺数量、主账号数量逐渐增多后，现有交互会出现三个问题：

1. 所有配置混在一张总表中，日、周、月配置难以快速区分
2. 当前配置对象实际已经是“店铺账号”，但页面仍缺少围绕主账号、区域、覆盖率的高效管理视角
3. 无法快速看出哪些店铺缺少日配置、周配置、月配置，存在漏采风险

同时，业务上已经明确：

- 最终采集配置对象应为“店铺账号”
- 主账号只作为分组、筛选、批量操作维度
- 同一店铺在同一粒度下允许存在多条启用配置
- 有头模式配置也算启用配置，且同样计入覆盖率

## 2. 设计目标

本次设计要实现以下目标：

1. 将采集配置明确拆分为日、周、月三个主视图
2. 明确“店铺账号”为最终配置对象
3. 让用户在选择店铺账号后自动套用该账号已保存的数据域能力
4. 对存在子数据域的数据域，默认自动全选子数据域
5. 保留手动调整，支持临时采集、排障测试和有头模式验证
6. 提供按日、按周、按月分别判断的覆盖率巡检能力
7. 在主账号和店铺规模上升后，依然能高效批量管理

## 3. 可选方案

### 3.1 方案 A：最小改造

只在现有采集配置页中新增：

- 日 / 周 / 月筛选
- 选择店铺账号后自动套用能力
- 未配置提醒

优点：

- 改动最小
- 上线快

缺点：

- 页面结构仍然是总表思路
- 主账号分组与批量管理能力不足
- 后续继续扩展时容易再次堆叠复杂度

### 3.2 方案 B：推荐方案

将采集配置改造成三层结构：

- 第一层：按粒度管理（日 / 周 / 月）
- 第二层：按平台、主账号、区域分组
- 第三层：按店铺账号配置与查看覆盖状态

同时补充：

- 自动套用店铺账号能力
- 子数据域自动全选
- 未覆盖 / 部分覆盖提醒
- 批量配置、复制配置、模板预留位

优点：

- 符合当前业务对象模型
- 兼顾手动灵活性与批量效率
- 能支撑后续主账号和店铺数量增长

缺点：

- 改动范围中等

### 3.3 方案 C：完全重构

拆成三个独立中心：

- 配置编辑中心
- 覆盖率巡检中心
- 模板中心

优点：

- 长期治理能力最强

缺点：

- 当前阶段实现成本过高
- 对现有用户习惯冲击较大

### 3.4 推荐结论

采用方案 B。

原因：

- 它保留了当前已经被验证有价值的“手动配置 + 临时采集 + 有头测试”模式
- 它将“店铺账号”作为最终配置对象，符合现有真实执行语义
- 它可以在不推翻现有功能的前提下，补足粒度分类、覆盖率巡检和规模化管理能力

## 4. 核心业务规则

### 4.1 配置对象

- 最终配置对象：店铺账号
- 上层组织维度：主账号
- 辅助筛选维度：平台、区域、店铺类型、数据域

### 4.2 粒度映射规则

以下时间选择统一映射到固定粒度：

- `今天`、`昨天`、`按日自定义` -> `daily`
- `最近7天`、`按周自定义` -> `weekly`
- `最近30天`、`按月自定义` -> `monthly`

这里的“按周自定义”“按月自定义”表示用户显式选择自定义时间范围时，同时显式声明其对应粒度。

### 4.3 覆盖率判断规则

覆盖率必须按粒度分别判断，不允许合并。

对单个店铺账号而言：

- 存在任意一条启用中的日配置 -> 日已覆盖
- 存在任意一条启用中的周配置 -> 周已覆盖
- 存在任意一条启用中的月配置 -> 月已覆盖

补充规则：

- 有头模式的启用配置，也算已覆盖
- 同一店铺、同一粒度下允许多条启用配置并存
- 覆盖率巡检只判断“是否至少存在一条启用配置”，不判断是否覆盖全部数据域

### 4.4 能力自动套用规则

用户选择店铺账号后：

- 自动读取该店铺账号在账号管理中保存的可采集数据域能力
- 自动勾选该店铺允许采集的数据域
- 如某数据域包含子数据域，默认自动全选全部子数据域

但保留用户手动覆盖：

- 用户可临时取消某些数据域
- 用户可临时取消部分子数据域
- 该次调整仅影响本次配置或任务，不强制回写店铺账号能力定义

## 5. 页面结构设计

### 5.1 页面入口

保留当前“采集配置管理”入口，但内部改为三视图：

- 日采集配置
- 周采集配置
- 月采集配置

默认进入最近使用的粒度视图，首次进入默认显示“日采集配置”。

### 5.2 每个粒度视图的结构

每个粒度视图内采用同一结构：

1. 顶部汇总条
2. 覆盖率提醒区
3. 主账号 / 区域 / 店铺筛选区
4. 当前粒度配置列表

### 5.3 顶部汇总条

展示：

- 当前粒度下总店铺数
- 已覆盖店铺数
- 未覆盖店铺数
- 部分覆盖店铺数

“部分覆盖”定义：

- 对整个系统三粒度视角而言，某店铺不是三类都齐全
- 在单粒度视图内可通过附加徽标显示其全局覆盖状态

### 5.4 覆盖率提醒区

在当前粒度视图中，显式展示：

- 当前缺少该粒度配置的店铺数量
- 缺失店铺清单入口
- 一键去补配入口

例如在“周采集配置”页中，只提醒“缺周配置”的店铺。

### 5.5 筛选与分组

支持以下筛选：

- 平台
- 主账号
- 区域
- 店铺类型
- 店铺账号
- 是否已覆盖当前粒度

支持以下分组展示：

- 平台 -> 主账号 -> 区域 -> 店铺账号

其中主账号和区域都应可折叠，便于大量店铺时快速浏览。

## 6. 配置创建与编辑流程

### 6.1 新建配置

流程如下：

1. 用户先选择当前粒度视图
2. 用户选择平台
3. 用户按主账号分组浏览店铺账号
4. 用户选择一个或多个店铺账号
5. 系统自动套用已保存能力
6. 系统自动勾选子数据域
7. 用户根据本次任务需要微调
8. 用户选择执行模式、调度规则等
9. 保存配置

### 6.2 编辑配置

编辑已有配置时：

- 保留该配置原始选择结果
- 可重新套用店铺账号能力
- 允许用户手动恢复“按账号能力推荐值”

### 6.3 手动模式保留

当前的手动配置方式必须保留。

它继续承担以下用途：

- 临时测试
- 单店铺排障
- 有头模式人工观察
- 针对某个数据域做局部验证

## 7. 覆盖率巡检能力

除了配置页中的提醒区，还需要后端提供店铺粒度覆盖率聚合接口。

该接口返回：

- 每个店铺账号在 daily / weekly / monthly 三个粒度下的覆盖状态
- 缺失粒度列表
- 是否部分覆盖
- 是否完全覆盖

配置页用它来实现：

- 当前粒度未覆盖列表
- 全局部分覆盖标记
- 一键跳转补配

## 8. 后端设计调整

### 8.1 现有配置模型

现有 `CollectionConfig` 已具备：

- `platform`
- `account_ids`
- `data_domains`
- `sub_domains`
- `granularity`
- `date_range_type`
- `execution_mode`
- `schedule_enabled`

本次不要求推翻现有模型，而是在此基础上明确其语义：

- `account_ids` 内的账号均视为店铺账号 ID
- `granularity` 成为覆盖率巡检的主判断字段

### 8.2 新增后端能力

新增或补强以下接口：

1. 店铺账号列表接口增强
2. 店铺账号覆盖率巡检接口
3. 主账号分组视图所需聚合接口
4. 批量补配辅助接口

建议新增接口：

- `GET /collection/config-coverage`

返回结构应至少包含：

- 店铺账号 ID
- 平台
- 主账号 ID
- 区域
- daily_covered
- weekly_covered
- monthly_covered
- missing_granularities

### 8.3 自动套用能力的数据来源

继续复用店铺账号能力来源：

- `ShopAccountCapability`

配置页选择店铺账号后：

- 直接根据该店铺账号能力映射生成推荐数据域与子数据域

## 9. 前端设计调整

### 9.1 `CollectionConfig.vue`

当前页面需要从“单页总表”升级为：

- 粒度页签
- 分组筛选
- 覆盖率提醒
- 店铺账号能力自动套用

### 9.2 账号选择器

当前选择器是普通多选下拉。

建议升级为：

- 支持主账号分组
- 支持区域过滤
- 支持显示该店铺已覆盖状态
- 支持显示该店铺能力摘要

### 9.3 保留快捷入口

保留：

- 快速配置
- 新建配置
- 手动执行

但快速配置需要改造成按粒度生成，而不是继续只面向总表。

## 10. 实施分期

### 10.1 第一阶段

第一阶段聚焦高价值闭环：

- 日 / 周 / 月三视图
- 粒度映射规则落地
- 店铺账号自动套用能力
- 覆盖率巡检接口
- 当前粒度缺失提醒

### 10.2 第二阶段

第二阶段提升效率：

- 主账号分组视图优化
- 批量补配
- 复制配置
- 模板能力

### 10.3 第三阶段

第三阶段做治理增强：

- 覆盖率巡检专页
- 更强的批量策略下发
- 模板版本化

## 11. 风险与边界

### 11.1 风险

- 旧配置中的粒度与日期选择可能存在历史不一致，需要统一归类
- 同一店铺同一粒度多条启用配置会增加列表阅读复杂度
- 若筛选与分组信息不足，用户仍会在大量店铺中难以定位

### 11.2 非目标

本次设计不直接解决：

- 是否要求“覆盖全部核心数据域”才算覆盖
- 是否对有头模式和无头模式做不同覆盖权重
- 是否引入复杂审批流或发布流

这些保持后续可扩展，但不进入本轮设计。

## 12. 结论

采集配置后续应围绕“店铺账号”组织，而不是回退到主账号对象。

推荐的升级方向是：

- 视图上按日 / 周 / 月拆分
- 组织上按主账号 / 区域分组
- 配置上自动套用店铺账号能力
- 治理上增加按粒度分别判断的覆盖率巡检
- 交互上继续保留手动配置和有头测试模式

这样既不会丢失当前已经很好用的灵活性，也能显著降低未来大量店铺场景下的配置和漏采风险。
