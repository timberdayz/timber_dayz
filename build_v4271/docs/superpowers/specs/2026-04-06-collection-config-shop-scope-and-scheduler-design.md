# 采集配置店铺维度与定时采集优化设计

## 背景

当前采集配置模型以全局 `account_ids + data_domains` 为中心，一条配置只能表达“这组店铺整体采哪些数据域”，不能表达“每个店铺采哪些数据域”。这会在多店铺场景下造成两个问题：

1. 页面上难以清晰表达每个店铺的实际采集范围。
2. 执行链路只能以全局数据域驱动，无法按店铺差异化拆任务。

与此同时，当前定时采集的配置保存与运行中调度器的 job 注册不是同一条主链路，导致页面状态与真实调度状态存在不一致风险。对于持续运行的采集环境，这种“保存后需重启才完全生效”的行为不可接受。

## 目标

1. 将采集配置升级为“单平台、多店铺、店铺维度数据域配置”模型。
2. 所有账号管理页中状态可用的店铺，都必须进入当前平台配置范围。
3. 每个店铺独立选择自己的 `data_domains / sub_domains`。
4. 定时采集在配置保存后立即同步到运行中的调度器。
5. 定时触发后按店铺拆任务，单店铺失败不阻断其他店铺执行。

## 非目标

1. 不做旧配置兼容迁移；测试阶段允许直接删除旧配置并用新模型重建。
2. 不支持一条配置跨多个平台。
3. 本期不扩展复杂补跑策略、分布式调度竞争控制和旧模型自动兼容。

## 核心约束

1. 一条配置只属于一个平台。
2. 平台下所有状态可用的店铺都必须在配置明细中出现。
3. 每个店铺至少选择一个数据域。
4. 店铺可选数据域受其能力约束。
5. 调度真源是数据库；APScheduler 仅为运行时投影。

## 数据模型设计

### 1. 配置头

保留 `CollectionConfig` 作为配置头，负责存储公共字段：

- `id`
- `name`
- `platform`
- `granularity`
- `date_range_type`
- `custom_date_start`
- `custom_date_end`
- `time_selection` 对应的统一时间选择语义
- `execution_mode`
- `schedule_enabled`
- `schedule_cron`
- `retry_count`
- `is_active`

配置头不再承担“全局 `account_ids` / 全局 `data_domains`”的表达职责。后续执行、展示和统计都以下游店铺明细为准。

### 2. 店铺明细

新增店铺维度明细模型，例如 `CollectionConfigShopScope`，字段建议如下：

- `id`
- `config_id`
- `shop_account_id`
- `data_domains`
- `sub_domains`
- `enabled`
- `created_at`
- `updated_at`

可选附加字段：

- `last_runtime_status`
- `last_error_message`

这些附加字段不是本期必需，但为后续观测保留扩展位。

### 3. 真源切换

后续所有运行链路以“配置头 + 店铺明细”作为唯一真源：

- 配置头定义公共规则。
- 店铺明细定义每店铺的实际采集内容。

## 页面交互设计

### 1. 信息层级

页面改为左右两层：

- 左侧：配置头公共信息
  - 配置名称
  - 平台
  - 粒度 / 日期范围
  - 执行模式
  - 定时设置
- 右侧：店铺维度配置列表
  - 每个店铺一个卡片或一行
  - 展示店铺基本信息、店铺能力、实际采集数据域、子类型选择

### 2. 交互原则

1. 不再维护全局 `data_domains` 作为最终配置。
2. “按店铺能力自动套用”变成逐店铺填充器，而不是对所有已选店铺求交集。
3. 所有可用店铺默认进入配置。
4. 保存前校验每个店铺至少一个数据域。

### 3. 批量操作

页面保留批量效率能力：

1. 按店铺能力自动套用。
2. 批量给多个店铺勾选某数据域。
3. 复制某店铺配置到其他店铺。

这些能力的本质是批量编辑店铺明细，而不是写回全局数据域集合。

## 调度与执行链路设计

### 1. 调度真源

数据库是唯一真源：

- `schedule_enabled / schedule_cron` 在数据库中定义事实状态。
- APScheduler 中的 job 必须与数据库保持一致。

### 2. 保存即同步

以下操作必须在保存成功后立即同步调度器：

1. 新建配置并启用定时：立即注册 job。
2. 修改 cron：立即重排 job。
3. 关闭定时：立即移除 job。
4. 删除配置：立即移除 job。

后端重启只负责兜底加载与对账，不再是主生效路径。

### 3. 定时触发

调度器命中 cron 后：

1. 读取配置头。
2. 读取该配置下的全部店铺明细。
3. 逐店铺创建 `CollectionTask`。
4. 每个店铺任务仅携带该店铺自己的 `data_domains / sub_domains`。

### 4. 容错规则

1. 单店铺失败不阻断其他店铺任务继续执行。
2. 单店铺无可采能力则跳过并记录原因。
3. 冲突检测保持店铺级粒度，而不是配置级全阻断。

### 5. 执行前二次校验

APScheduler 触发前或触发后实际执行前，必须再次校验：

1. 配置存在。
2. `is_active = true`
3. `schedule_enabled = true`
4. `schedule_cron` 仍有效
5. 店铺仍处于可用状态

防止数据库关闭定时后，内存中旧 job 继续误触发。

## 接口设计变化

### 配置接口

- `GET /collection/configs`
  - 返回配置列表摘要
  - 包含店铺数、定时状态、最近执行状态
- `GET /collection/configs/{id}`
  - 返回配置头 + `shop_scopes`
- `POST /collection/configs`
  - 创建配置头和店铺明细
- `PUT /collection/configs/{id}`
  - 更新配置头和店铺明细
- `DELETE /collection/configs/{id}`
  - 删除配置并移除 job

### 调度接口

保留 `POST /collection/configs/{id}/schedule`，但其语义应收口为统一调度同步入口，供后端内部或前端独立操作使用。推荐实际保存配置时由后端统一调用调度同步逻辑，而不是依赖前端二次发请求保证一致性。

## 保存校验规则

后端保存配置时必须完成以下校验：

1. 只允许单平台。
2. 平台下所有可用店铺必须都出现在 `shop_scopes`。
3. 每个店铺至少一个数据域。
4. 每个店铺的数据域必须受店铺能力约束。
5. 子类型必须隶属于该店铺已选主数据域。
6. 启用定时时 cron 表达式必须合法。
7. 若定时开启，则数据库保存与调度器同步必须作为一个完整成功结果；若调度同步失败，保存应失败并回滚。

## 手动执行与定时执行统一

手动“执行配置”和定时触发应共用同一套“按店铺拆任务”的服务逻辑：

1. 读取配置头和店铺明细。
2. 逐店铺创建任务。
3. 逐任务按店铺维度能力过滤并执行。

避免手动执行与定时执行出现两套口径。

## 可观测性

建议配置详情与列表补充以下调度状态：

1. `next_run_time`
2. `last_run_time`
3. `last_run_status`
4. `job_registered`

这样页面上的“启用定时”不再只是数据库开关，而是可核对的运行时事实。

## 删除旧配置策略

当前仍处于测试阶段，允许直接删除旧配置。上线新模型前可执行一次清理：

1. 清空旧采集配置。
2. 清空相关旧定时 job。
3. 用新页面手工重建测试配置。

因此本期实现不包含旧配置迁移逻辑。

## 测试与验收范围

### 功能验收

1. 可按店铺独立选择数据域。
2. 所有可用店铺默认进入配置范围。
3. 每个店铺至少一个数据域。
4. 保存配置后定时立即生效。
5. 修改 cron 后立即生效。
6. 关闭定时后立即失效。
7. 删除配置后 job 被移除。
8. 手动执行与定时执行均按店铺维度拆任务。
9. 单店失败不阻断其他店铺。

### 测试建议

1. Schema / API 合约测试
2. 配置保存与回显测试
3. 调度注册 / 更新 / 删除测试
4. 重启后调度恢复与对账测试
5. 手动执行与定时执行的店铺级拆任务测试

## 分期建议

### 第一期

1. 新模型落库
2. 页面改成店铺维度
3. 保存即同步调度器
4. 手动执行与定时执行按店铺拆任务

### 第二期

1. 更细的调度观测
2. 店铺失败统计
3. 店铺级历史运行态展示

## 结论

本设计将采集配置从“全局数据域配置”升级为“单平台、多店铺、店铺维度数据域配置”，并将定时采集从“数据库状态与调度状态可能分离”升级为“保存即同步、数据库为真源、按店铺独立容错”的运行模型。这一方案能直接解决多店铺配置表达不清和持续运行采集环境中调度不一致的问题，同时控制了测试阶段的实现范围，不引入旧配置迁移和跨平台复杂度。
