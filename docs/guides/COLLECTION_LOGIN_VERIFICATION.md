# 数据采集：登录验证码处理

本文档说明采集任务中**验证码类型、检测时机、人工回填与可选打码**的契约，以及有头/无头一致、超时与部署约束。

## 验证码类型

| 类型 | 说明 | verification_type | 前端展示 |
|------|------|-------------------|----------|
| 图形验证码 | 登录页或导出页出现的图片验证码 | `graphical_captcha` | 截图 + 输入框 |
| 短信/邮箱/OTP | 二次验证、短信/邮箱验证码 | `otp`（或 `sms`/`email_code`） | 输入框（无截图） |

## 检测时机

- **登录阶段**：点击「登录」按钮后，**约 2–3 秒**内检查是否仍停留在登录页且出现验证码相关 DOM（如「请输入验证码」文案、验证码输入框）。若检测到则截图并进入「需要验证码」流程，**不做长时间轮询**。
- **导出阶段**：某数据域导出时若页面出现验证码，同样检测并进入同一套「暂停 → 回传 → 同一 page 填入 → 继续」流程。
- 检测逻辑**有头与无头完全相同**，仅依赖 DOM/URL/文案，与浏览器是否可见无关；应对流程按**无头生产**设计（人工回填为主、打码可选）。

## 人工回填（主路径）

1. 执行器检测到验证码后，通过**回调**将 `verification_type`、`verification_screenshot` 写入任务表，任务状态置为 `paused`。
2. 前端通过**轮询**任务列表/详情发现 `status === 'paused'` 且存在 `verification_type`，展示验证码弹窗（截图来自 `GET /tasks/{task_id}/screenshot`，或任务返回的 `verification_screenshot` 信息）。
3. 用户输入验证码或 OTP 后提交，前端调用 `POST /tasks/{task_id}/resume`，请求体为 `{ "captcha_code": "..." }`（图形验证码）或 `{ "otp": "..." }`（短信/邮箱验证码）。
4. 后端将提交内容写入 **Redis**（key：`collection:captcha:{task_id}`），**正在等待中的执行器**（同一进程、同一 page）轮询到后在同一 page 上填入并点击提交，然后继续执行。
5. **恢复形态**：**同一进程、同一页面内等待回传后继续**；不采用「paused 后进程退出再重新启动任务」，因刷新/重开页面会使当次验证失效。

## 超时与失败

- 等待用户回传的**超时时间**由环境变量 `VERIFICATION_TIMEOUT` 控制（默认 300 秒，约 5 分钟）。
- 超时后执行器结束等待，将任务置为**失败**（或专用状态），并释放浏览器。
- 超时或任务已结束后，用户再调用 Resume 提交验证码时，接口返回 **4xx**，提示「验证已超时或任务已结束，请重新发起采集」。

## 可选：第三方打码服务

- 若产品接入 2Captcha 等打码 API，可在检测到图形验证码时截取图片并调用 API，将识别结果填入；**人工回填为默认或必选能力**，打码为可选增强。
- 使用打码服务时需在配置或文档中说明使用场景与数据合规，默认关闭。

## 多实例与部署约束

- 验证码**截图**若为单机路径，多实例部署时需将截图写入**共享存储**（如 NFS、对象存储），或约定验证码能力仅在单实例或「任务与实例绑定」部署下可用。
- 验证码等待期间**执行该任务的进程须保持存活**；滚动重启、扩缩容可能导致等待中的任务被终止，用户提交的验证码将无效。部署时需评估等待超时与重启策略的兼容性。

## 相关 API

- `GET /collection/tasks/{task_id}`：任务详情，含 `verification_type`、`verification_screenshot`。
- `GET /collection/tasks/{task_id}/screenshot`：获取验证码截图图片（读取任务表 `verification_screenshot` 对应文件）。
- `POST /collection/tasks/{task_id}/resume`：提交验证码并恢复。请求体 `{ "captcha_code": "..." }` 或 `{ "otp": "..." }`；仅当任务为 `paused` 时有效。
