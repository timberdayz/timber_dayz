# 采集模块架构原则设计

日期：2026-03-18

## 背景

当前仓库中的采集体系已经具备以下主干能力：

- 录制器负责采集用户操作步骤
- 生成器将步骤转换为 Python 组件
- 组件通过版本管理保存
- 执行器按平台和数据域加载并运行组件
- 运行时统一处理验证码、弹窗、会话、下载、重试和可观测性

这说明系统已经不再适合继续沿用“录一段脚本，直接回放上线”的旧思路。接下来需要明确一组长期稳定的架构原则，避免新旧路径混杂、录制产物失控、运行时职责下沉到脚本内部。

## 目标

本设计只回答一个问题：

如何让“录制 -> 生成 -> 人工微调 -> 按平台和数据域执行采集”成为唯一可信、可维护、可扩展的主路径。

## 备选路线

### 方案 A：录制脚本中心化

做法：

- 录制产出直接作为执行脚本
- 失败时主要靠人工重录
- 执行器尽量少做约束

优点：

- 上手快
- 初期出成果快

缺点：

- 录制产物天然脆弱
- 难以复用平台级能力
- 账号、验证码、会话、下载、异常处理容易散落在脚本里
- 一旦平台改版，维护成本会迅速失控

结论：

不推荐，最多只能作为调试和临时诊断路径。

### 方案 B：组件中心化生成链路

做法：

- 录制只产出步骤
- 步骤统一进入生成器
- 生成器产出符合组件契约的异步 Python 组件
- 人工只在组件层微调
- 执行器负责运行时能力

优点：

- 录制、代码资产、运行时边界清晰
- 更适合平台化与数据域复用
- 便于版本管理、测试、回归和审计

缺点：

- 前期需要严格治理生成器和运行时边界
- 需要持续清理旧资产和旧流程

结论：

这是推荐的主路径。

### 方案 C：API 优先，UI 导出兜底

做法：

- 在方案 B 基础上，数据获取优先走稳定 API
- 页面点击、导出、下载作为兜底能力

优点：

- 稳定性和执行效率更高
- 对页面改版敏感度更低

缺点：

- 不是所有平台、所有数据域都存在可用 API
- 仍然需要 UI 自动化能力兜底

结论：

它不是对方案 B 的替代，而是方案 B 之上的数据获取策略。

## 推荐设计

推荐采用“方案 B 为主，方案 C 为增强”的结构：

1. 录制是输入，不是生产资产。
2. 组件是唯一进入正式执行链路的脚本资产。
3. 平台和数据域是运行时装配单元，不是录制时的临时标签。
4. 执行器统一承接所有跨组件的运行时能力。
5. API 优先、UI 兜底是数据获取策略，不改变主架构边界。

## 五条必须坚持的架构原则

### 原则 1：录制产物不是可执行真相源

录制结果只能作为原始输入或诊断材料，不能直接视为生产脚本。

强约束：

- 原始录制文件不得直接进入正式采集链路
- 原始录制文件不得承载账号、密码、环境配置、下载路径等生产信息
- 录制产物必须先经过规范化和组件生成，才能进入保存和测试阶段

### 原则 2：生成器只负责“步骤转组件”，不负责“运行时智能”

生成器的职责是把录制步骤转成符合契约的组件代码，而不是在代码里塞满执行器逻辑。

生成器应该负责：

- 组件骨架
- 步骤翻译
- 组件元数据
- 容器上下文
- 必要的成功条件模板

生成器不应该负责：

- 泛化重试框架
- 全局弹窗治理
- 会话复用策略
- 平台级验证码协调
- 大量运行时门卫检测

这些都应该由执行器、适配层或平台公共组件承担。

### 原则 3：平台是能力边界，数据域是执行单元

平台负责承载登录、导航、会话、指纹、平台级弹窗和公共组件能力；数据域负责承载具体的导出目标、选择器配置和落盘约束。

强约束：

- 不允许把平台共性逻辑重复写进每个数据域组件
- 不允许让数据域名只存在于页面脚本注释里，必须进入正式元数据和调度参数
- 调度、前端、版本管理、执行器都应以 `platform + component_type + data_domain + sub_domain` 为稳定主键语义

### 原则 4：运行时稳定性必须外置，不下沉到录制脚本内部

验证码、弹窗、会话恢复、重试、下载管理、步骤可观测性、错误分类、任务恢复，必须由运行时统一承接。

理由：

- 这些能力是跨组件共享的
- 它们需要全局一致性
- 它们依赖任务上下文，而不只是单页步骤

判定标准：

- 如果一个能力在多个平台/多个组件重复出现，就应该优先外置
- 如果一个能力需要任务级状态，就不该放进生成器默认模板

### 原则 5：仓库里只能有一条正式生成主路径

允许保留历史资产，但不能允许多条“看起来都能生成采集脚本”的正式路线长期并存。

强约束：

- 必须明确哪条链路是正式主路径
- 旧模板生成器、旧回放脚本、旧录制文件只能作为历史参考或诊断工具
- 正式保存、测试、版本化、执行，只能围绕一条 canonical path 展开

当前建议的 canonical path：

`Inspector 录制 -> steps -> steps_to_python -> Python 组件 -> 版本管理 -> 执行器运行`

## 禁止事项

以下做法应视为架构违规：

- 把原始录制脚本直接当生产采集脚本
- 在录制文件或样例脚本中保留明文凭据
- 让同步 Playwright 回放脚本进入正式异步执行链路
- 在多个目录同时维护“新的正式生成器”
- 把重试、验证码、全局弹窗、会话恢复分别散落到每个数据域组件
- 让前端、执行器、版本管理对同一个组件使用不同命名语义

## 近期治理重点

### 第一优先级

- 明确并公开唯一正式生成主路径
- 把旧模板生成器和原始回放脚本明确标记为历史/调试用途
- 清理或隔离包含敏感信息的旧录制资产

### 第二优先级

- 继续补齐组件版本管理回归与验收
- 补齐测试环境与生产环境的执行一致性
- 固化步骤可观测性，保证故障能定位到阶段、组件和数据域

### 第三优先级

- 在平台层沉淀更多可复用能力，减少各数据域组件局部启发式代码
- 对稳定数据域逐步推进 API 优先、UI 兜底

## 结论

这套采集设计的方向是正确的，问题不在“录制 + 生成 + 微调 + 执行”这个思路本身，而在于是否能把它严格收敛成一条唯一主路径，并把运行时稳定性能力从录制产物中剥离出来。

只要坚持上面的五条原则，这个体系可以继续演进；如果放任旧路径并存、录制产物直接入生产、运行时逻辑散落在脚本里，后续维护成本会持续上升。
