# Collection Module Boundaries Design

**日期:** 2026-03-31

**目标:** 收紧“采集配置 / 采集任务 / 采集历史”三个页面的职责边界，避免重复造轮子，并支持多账号并发采集时的验证码回填与人工有头干预。

## 1. 背景

当前采集模块已经具备：

- 配置驱动创建任务
- 手动任务执行
- 定时任务执行
- 任务级验证码恢复

但页面职责仍然存在交叉：

- `采集配置` 既负责模板定义，又直接触发执行
- `采集任务` 负责运行态，但配置页没有完整接入任务恢复能力
- `采集历史` 与任务列表在终态展示上存在部分语义重叠

多账号并发采集时，这种边界不清会放大问题：

- 多个账号同时需要验证码回填
- 多个账号同时需要切换有头模式人工处理
- 用户无法明确在哪个页面处理这些阻塞项

## 2. 设计目标

本设计要实现以下目标：

1. 明确三个页面的职责边界
2. 验证码回填和人工干预只保留一套实现
3. 支持多账号并发采集时的统一待处理队列
4. 保留配置驱动定时任务和批量任务的能力
5. 不把运行时交互重新塞回配置页

## 3. 页面职责

### 3.1 采集配置

`采集配置` 是模板层与策略层。

负责：

- 平台、账号范围、数据域、时间规则配置
- 默认执行模式配置：`headless` / `headed`
- 定时规则配置
- 批量创建任务

不负责：

- 验证码输入
- 运行中日志查看
- 人工干预面板
- 任务恢复状态机

点击“执行”后的标准行为：

1. 按配置生成 1 个或多个任务
2. 创建成功后跳转到 `采集任务`
3. 将本次任务作为默认过滤条件带入任务页

### 3.2 采集任务

`采集任务` 是执行层与干预层。

负责：

- 手动创建单个任务
- 运行中任务列表
- 任务实时状态、日志、进度
- 验证码回填
- 人工干预入口
- 取消、重试、继续执行

这里是整个采集模块唯一的“运行态交互中心”。

### 3.3 采集历史

`采集历史` 是归档层与审计层。

负责：

- 终态任务查询
- 历史结果追溯
- 统计与审计

不负责：

- 验证码处理
- 运行中任务控制
- 人工干预

## 4. 状态模型

三个页面围绕同一套任务状态工作，但消费不同子集。

完整任务状态建议至少包括：

- `pending`
- `running`
- `verification_required`
- `manual_intervention_required`
- `verification_submitted`
- `paused`
- `completed`
- `failed`
- `cancelled`
- `partial_success`

页面关注范围：

- `采集配置`：只关心配置级摘要与是否存在阻塞任务
- `采集任务`：关心完整运行态
- `采集历史`：只关心终态

## 5. 多账号并发处理

多账号并发时，`采集任务` 页必须支持“待处理队列”。

建议拆成两个视图：

### 5.1 待处理队列

用于展示所有需要人工动作的任务：

- 图形验证码
- OTP
- 滑块
- 需要切换有头模式的人工干预

每个待处理项至少显示：

- 平台
- 账号
- 来源配置
- 当前阻塞类型
- 进入阻塞时间

### 5.2 普通任务列表

展示：

- pending
- running
- completed
- failed
- cancelled
- partial_success

这样可以避免多个账号同时阻塞时，用户不知道先处理哪个。

## 6. 执行模式设计

执行模式应该作为“配置默认值 + 任务可覆盖值”。

建议字段：

- `execution_mode: headless | headed`

或继续沿用现有语义：

- `debug_mode: true | false`

设计原则：

- `采集配置` 定义默认执行模式
- 配置触发的手动任务继承该模式
- 定时任务继承该模式
- `采集任务` 页手动新建任务时可覆盖该模式

同时，“切换有头模式人工处理”应当是任务级恢复动作之一，而不是配置页单独实现的一套能力。

## 7. 验证码与人工干预

验证码恢复与人工干预只能保留一套实现，归属于 `采集任务`。

### 7.1 验证码

支持：

- 图形验证码提交
- OTP 提交

### 7.2 人工干预

支持：

- 打开有头模式处理
- 处理完成后继续当前任务

配置页不直接嵌入验证码或人工干预面板，只在创建任务后引导进入任务页处理。

## 8. 页面交互流

标准交互流：

1. 用户在 `采集配置` 中点击执行
2. 系统批量创建任务
3. 页面自动跳转 `采集任务`
4. 如果部分任务进入验证码或人工干预状态，统一出现在待处理队列
5. 任务完成后，结果沉淀到 `采集历史`

## 9. 接口边界

本设计不建议为配置页重复发明一套验证码恢复接口。

### 9.1 配置层接口

- `POST /collection/configs`
- `PUT /collection/configs/{id}`
- 配置驱动的批量创建任务接口

### 9.2 任务层接口

- `GET /collection/tasks`
- `GET /collection/tasks/verification-items`
- `POST /collection/tasks/{task_id}/resume`
- 后续可扩展：
  - `POST /collection/tasks/{task_id}/switch-headed`
  - `POST /collection/tasks/{task_id}/manual-complete`

### 9.3 历史层接口

- `GET /collection/history`

## 10. 非目标

本设计不包含：

- 重写采集组件执行逻辑
- 重写 runtime resolver
- 删除现有三个页面

它只解决页面职责和交互边界问题。

## 11. 最终结论

这三个页面不是重复造轮子，而是边界还未完全收口。

正确方向不是删除页面，而是明确三层职责：

- `采集配置`：模板、调度、批量发起、默认执行模式
- `采集任务`：运行、验证码、人工干预、重试、取消
- `采集历史`：终态查询、归档、审计

本设计推荐：

- 保留三页
- 共用一套任务状态机
- 共用一套验证码/人工干预能力
- 让配置页成为任务生成器，而不是第二个任务控制台
