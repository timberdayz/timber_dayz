# Login Entry De-Shopification Design

**Date:** 2026-04-05

**Goal:** Remove shop-specific login entry behavior from main-account runtime configuration so formal collection always enters a platform-neutral authenticated shell before switching to the target shop.

## 1. Background

The current collection runtime has already been moved to main-account-scoped session ownership. That solved one class of session duplication problems, but another legacy assumption remains:

- `main_account.login_url` can still contain shop-specific parameters
- runtime login may therefore begin from a concrete store context
- under same-main-account concurrent collection, multiple tasks can inherit the wrong starting shop context before explicit shop switching begins

That is architecturally wrong.

Main accounts should represent authentication identity, not shop selection.

## 2. Goal

This phase should make the runtime rule single and explicit:

> Formal collection login entry is generated by platform runtime rules, not by database-stored shop-bound URLs.

In the test environment, this should be handled directly and cleanly rather than preserved behind compatibility branches.

## 3. Core Rule

For formal collection:

- do not trust `main_account.login_url` as the runtime source of truth
- always use a platform-defined, shop-neutral login entry
- always perform shop selection after login

This applies regardless of whether old database rows still contain a `login_url`.

## 4. Ownership Model

### 4.1 Main Account

Main account should own only:

- username
- password or encrypted credential
- platform
- main account identity

It should not own final shop context.

### 4.2 Shop Account

Shop account should own:

- target platform shop identifier
- region
- store context metadata

### 4.3 Runtime

Runtime should own:

- platform login entry generation
- login shell readiness checks
- post-login shop switching

## 5. Platform Login Entry Strategy

Each platform should expose a canonical login-entry generator.

Recommended model:

- `shopee` -> fixed shop-neutral seller login entry
- `miaoshou` -> fixed platform-neutral login entry
- `tiktok` -> fixed seller-center login entry

These entries should contain authentication-routing parameters only.

They must not contain:

- `shop_id`
- `cnsc_shop_id`
- region-bound store query state
- store-specific redirect targets

## 6. Runtime Behavior

Formal collection execution should use this order:

1. generate platform-neutral login entry
2. enter authenticated platform shell
3. coordinate shared-state preparation by main account
4. switch to target shop using shop-account data
5. confirm target shop is ready
6. begin collection

This guarantees that login and shop targeting are no longer conflated.

## 7. Database Strategy

Since this is currently a test environment, the recommendation is to handle it directly rather than keep redundant compatibility logic.

### 7.1 Data Cleanup

Clean all existing `main_account.login_url` values so they no longer contain shop-specific parameters.

### 7.2 Interface Enforcement

Even if the field remains in the schema, create/update flows should no longer allow shop-bound URLs to persist.

In practice:

- incoming login URLs should be normalized to platform-neutral entries
- or overwritten entirely by platform-standard values

## 8. Admin UI Strategy

The admin UI should stop implying that login URL is a freely editable shop-bound runtime field.

Recommended options:

- hide the field entirely
- or render it as a read-only platform-standard value

Do not keep a misleading editable login URL input that suggests operators should paste shop-specific seller links.

## 9. Scope

This phase should include:

- platform-neutral login entry generation
- runtime switch away from database-driven login URLs
- data cleanup for existing `main_account.login_url` values
- interface enforcement preventing reintroduction of shop-bound URLs
- admin-UI cleanup if needed

This phase should not include:

- unrelated account-management redesign
- collection-config model changes
- scheduler redesign
- template system work

## 10. Rollout Strategy

Implement in this order:

1. add platform login-entry generator(s)
2. switch formal collection runtime to use them
3. add backend normalization/overwrite logic for main-account writes
4. clean existing database rows
5. simplify admin UI treatment of login URL

## 11. Testing Strategy

### Backend

Add tests proving:

- runtime login entry is platform-generated rather than copied from DB
- shop-bound login URLs are normalized away on write
- shop switching still uses shop-account data after login

### Manual

Validate:

- same-main-account multi-shop Shopee collection
- same-main-account multi-shop Miaoshou collection
- same-main-account multi-shop TikTok collection

## 12. Recommendation

The recommended approach is the strict one:

- runtime does not consume DB-stored shop-bound login URLs
- database values are cleaned
- interface writes are normalized
- shop selection is always post-login

That keeps the model clean and removes a whole category of future concurrency bugs.
