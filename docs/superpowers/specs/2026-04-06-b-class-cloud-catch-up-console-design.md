# B类数据云端追平控制台设计

**Date:** 2026-04-06
**Status:** Draft for user review

## Goal

将当前“云端同步管理”页面从表级操作台重构为一个面向中文管理员的 **B类数据云端追平控制台**。

新页面的核心目标不是让管理员理解或操作单张表，而是让管理员能够快速判断：

1. 云端 B 类数据是否已追到最新
2. 自动同步是否开启且工作正常
3. 当前是否存在积压、失败或部分成功
4. 现在最应该执行的动作是什么

## Product Positioning

新页面的产品定位为：

**采集环境到云端的 B 类数据追平控制台**

它服务于以下真实运行模型：

1. 采集环境按计划执行数据采集
2. 采集完成后，本地进行清洗和写入 `b_class.fact_*`
3. 云同步系统将本地 B 类数据追平到云端
4. 管理员只关心整体是否最新、是否有异常、是否需要干预

因此，页面主心智从：

- “我要操作哪张表”

调整为：

- “我要确保云端 B 类数据保持最新”

## Non-Goals

本次页面重构不追求：

- 将首页继续做成表级技术操作面板
- 暴露所有底层技术字段给非技术管理员
- 让管理员通过首页逐表执行同步
- 让首页承担数据库修复工具页的职责

这些能力如果仍有保留价值，应下沉为高级诊断能力。

## Target Data Scope

当前控制台管理的同步范围限定为：

- 本地采集环境 `b_class` schema 下的 `fact_*` 表
- 对应的数据域包括：
  - `orders`
  - `products`
  - `analytics`
  - `services`
  - `inventory`

当前控制台不负责：

- `semantic`
- `mart`
- `api`
- 非 B 类数据

## Core Interaction Model

页面主操作模型收敛为 2 种模式 + 1 个异常恢复动作。

### 1. 自动同步

自动同步是主模式。

语义：

- 当采集完成且本地 B 类数据清洗入库成功后，系统自动将本地与云端之间的落后部分追平
- 当管理员暂停自动同步时，系统继续记录待同步内容，但不执行
- 恢复自动同步后，系统自动补齐暂停期间积压的数据

页面只需要表达：

- 当前自动同步是否开启
- 当前自动同步是否工作正常
- 暂停后是否会自动补齐

### 2. 立即同步到最新

这是管理员兜底动作，不是首页主路径。

语义：

- 对当前落后部分执行一次增量追平
- 不是“选定一张表同步”
- 不是“全量重跑历史数据”

页面按钮文案：

- `立即同步到最新`

确认文案：

- `将立即检查本地与云端B类数据差异，并对未同步部分执行追平。是否继续？`

### 3. 重试异常任务

这不是第三种模式，而是恢复动作。

语义：

- 当系统中存在 `失败` 或 `部分成功` 的任务时，允许管理员触发统一重试
- 管理员不需要先逐条理解或逐条点击大多数异常任务

## Information Architecture

首页采用四层信息架构。

### Layer 1: 总览层

回答“现在整体是否健康、是否最新”。

建议展示以下状态卡：

- 自动同步
- 同步服务
- 云端连接
- 云端状态
- 异常任务
- 上次成功同步

### Layer 2: 主操作层

回答“管理员现在应该做什么”。

只保留以下主操作：

- `立即同步到最新`
- `开启自动同步 / 暂停自动同步`
- `重试异常任务`

不再在首页主区域展示：

- 表选择
- Batch Size
- 单表 Dry Run
- Repair Checkpoint
- Refresh Projection

### Layer 3: 运行反馈层

回答“系统最近做了什么、现在做到哪一步”。

建议展示：

- 当前是否正在追平
- 当前批次编号
- 当前阶段
- 已处理 / 待处理
- 最近错误摘要
- 最近几次同步结果

### Layer 4: 高级诊断层

回答“如果出问题了，我怎么查”。

下沉能力包括：

- 表级状态
- checkpoint 明细
- 失败任务列表
- 单表 dry-run
- 单表 checkpoint 修复
- projection 刷新
- lease / heartbeat / retry 时间等恢复字段

## Homepage Copy Direction

页面名称建议改为：

- `B类数据云端追平控制台`

副标题建议：

- `用于管理采集环境到云端的B类数据同步，确保云端始终保持最新状态。`

主状态文案统一为中文业务语义，而不是直接暴露英文底层状态词。

建议使用如下中文映射：

- 自动同步：`已开启` / `已暂停`
- 同步服务：`运行中` / `未启动` / `配置异常`
- 云端连接：`连接正常` / `连接异常`
- 云端状态：`已追平最新` / `正在追平` / `存在积压`
- 执行结果：`成功` / `部分成功` / `失败`

## Homepage Structure

### 1. 标题区

标题：

- `B类数据云端追平控制台`

副标题：

- `用于管理采集环境到云端的B类数据同步，确保云端始终保持最新状态。`

右侧保留：

- `刷新状态`

若存在异常任务，在标题区增加醒目告警：

- `当前有 N 个异常同步任务待处理`

### 2. 总览状态区

以 6 张状态卡表达整体状态：

- 自动同步
- 同步服务
- 云端连接
- 云端状态
- 异常任务
- 上次成功同步

目标是让管理员在 5 秒内完成判断：

- 是否正常
- 是否最新
- 是否需要干预

### 3. 主操作区

单独做成“控制面板”。

主操作包括：

- `立即同步到最新`
- `开启自动同步 / 暂停自动同步`
- `重试异常任务`

自动同步说明文案：

- 开启时：`采集完成并完成本地清洗入库后，系统将自动追平云端B类数据`
- 暂停时：`暂停期间仍记录待同步内容，恢复后自动补齐`

### 4. 运行反馈区

分为两块：

- `最近同步状态`
- `最近同步记录`

最近同步状态包括：

- 当前状态
- 当前批次
- 当前阶段
- 已处理 / 待处理
- 最近错误摘要

最近同步记录包括：

- 发起方式：自动 / 手动 / 补偿
- 开始时间
- 结束时间
- 结果
- 异常数
- 摘要

### 5. 高级诊断区

作为折叠区或次级入口存在。

用于承接所有技术性、低频、故障定位型能力。

## Frontend Behavior Changes

### Remove From Primary Workflow

以下能力不再占据首页主区域：

- `Source Table`
- `Batch Size`
- `Dry Run`
- `Repair Checkpoint`
- `Refresh Projection`

### Keep As Advanced Diagnostics

以下能力仍可保留，但下沉：

- 表级同步状态
- 单表任务查看
- 单表 checkpoint 修复
- 单表 dry-run
- 单表 projection 刷新
- 原始错误明细

### Empty State

首页需要新增清晰的空状态和提示，不再让管理员面对没有上下文的表格。

示例：

- `当前尚无待同步内容`
- `自动同步已开启，采集完成后将自动追平云端`
- `当前没有异常任务`

## Backend Contract Changes

为了支撑“追平控制台”而不是“单表操作台”，建议新增或重组以下控制面接口。

### Primary APIs

#### `GET /api/cloud-sync/overview`

返回：

- 自动同步状态
- 同步服务状态
- 云端连接状态
- 是否已追平最新
- 异常任务数
- 上次成功同步时间
- 最近错误摘要

#### `POST /api/cloud-sync/sync-now`

语义：

- 对当前落后部分执行一次增量追平
- 不要求用户选择单表

#### `GET /api/cloud-sync/runtime`

返回：

- 当前是否正在执行
- 当前批次编号
- 当前阶段
- 已处理数 / 待处理数
- 当前开始时间

#### `GET /api/cloud-sync/history`

返回最近同步记录：

- 发起方式
- 开始时间
- 结束时间
- 结果
- 异常数
- 摘要

#### `GET /api/cloud-sync/settings`

返回自动同步配置和状态。

#### `PUT /api/cloud-sync/settings`

用于：

- 开启自动同步
- 暂停自动同步

#### `POST /api/cloud-sync/retry-failed`

用于统一重试当前失败或部分成功任务。

### Advanced APIs

现有以下接口保留，但降级为高级诊断接口：

- `GET /api/cloud-sync/tables`
- `GET /api/cloud-sync/tasks`
- `GET /api/cloud-sync/tasks/{job_id}`
- `POST /api/cloud-sync/tables/{table_name}/dry-run`
- `POST /api/cloud-sync/tables/{table_name}/repair-checkpoint`
- `POST /api/cloud-sync/tables/{table_name}/refresh-projection`

## State And Safety Rules

### 1. Pause Behavior

当自动同步暂停时：

- 系统继续记录待同步内容
- 不执行同步
- 恢复后自动补齐

### 2. Manual Sync Behavior

当管理员点击 `立即同步到最新` 时：

- 系统只补齐落后部分
- 不触发历史全量重跑

### 3. Retry Behavior

`重试异常任务` 只作用于：

- `failed`
- `partial_success`

### 4. Admin Safety

首页所有动作都应保留二次确认和明确反馈。

例如：

- `已提交追平任务`
- `自动同步已暂停`
- `异常任务已提交重试`

## Rollout Strategy

建议直接重构现有 `/cloud-sync` 页面，而不是新增第二个同步页面。

实施顺序：

1. 先改首页信息架构和中文文案
2. 再改主操作区
3. 再补 `overview/runtime/history/settings/retry-failed` 接口
4. 最后把旧表级能力下沉到高级诊断

这样可以避免：

- 菜单重复
- 新旧两套入口并存
- 管理员不知道该用哪个页面

## Acceptance Criteria

### Product Acceptance

管理员进入页面后，能在 5 秒内知道：

1. 自动同步是否开启
2. 云端是否已追到最新
3. 当前是否有异常
4. 现在该做什么

### Functional Acceptance

- 点击 `立即同步到最新` 会触发增量追平，而不是要求选表
- 暂停自动同步后，新数据仍会形成待同步内容
- 恢复自动同步后，会自动补齐暂停期间积压

### Operational Acceptance

- 失败和部分成功在首页清晰可见
- 高级诊断中能看到故障恢复所需字段
- 首页不会要求管理员理解底层表结构才能完成主操作

### UX Acceptance

- 页面主文案全中文
- 首页不混用英文技术词作为主展示文案
- 首页不再以单表操作作为主视觉
- 桌面端信息层级清楚，移动端不破版

## Open Implementation Notes

1. 现有表级接口仍有保留价值，但不应继续主导首页产品结构。
2. 当前 runtime 启动方式与“专用 worker 角色”仍存在差距，后续实现阶段需要一起校正。
3. 首页的“已追平最新”状态需要后端提供一个可靠的总体判定逻辑，而不是让前端拼接表级状态自行推断。

## Recommendation

按本设计将当前页面重构为：

**中文、整体状态优先、自动同步为主、手动增量追平为辅、表级能力下沉的 B 类数据云端追平控制台。**
