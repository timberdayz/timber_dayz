# Monthly Profit Settlement Center Design

## Goal

为西虹 ERP 新增“月度利润结算中心”，把当前分散在店铺利润基准、人员提成、工资单、跟投收益中的月度经营结果，统一汇总为公司级月结总账。

本设计的目标不是替换现有 HR、薪资、跟投、店铺提成模块，而是：

- 将它们重新定义为月结中心的上游来源模块
- 建立“公司-月份”唯一口径的月结主表
- 让公司可以直观看到某个月：
  - 公司月度净利润
  - 人员成本总额
  - 跟投收益总额
  - 公司留存总额
- 支持“目标比例 + 实际结算 + 差异审批”的经营模式

## Why This Is Needed

当前系统已经具备若干局部能力，但没有统一的公司月结口径：

- 店铺级结算基准利润：`profit_basis_amount`
- 店铺级人员提成配置：人员归属、提成比例、可分配净利润率
- 员工工资单：固定薪资、绩效工资、奖金、社保、公积金、个税等
- 店铺级跟投收益试算与审批

这些能力分别可用，但尚未统一回答一个经营问题：

```text
本月净利润 100%，最终分成了多少人员成本、多少跟投收益、多少公司留存？
```

因此需要新增一个“月度利润结算中心”作为公司级总账与审批入口。

## Design Principles

### 1. 不推翻现有模块

第一阶段不重写现有 HR、工资单、跟投、店铺提成逻辑。

改造原则：

- 现有模块继续负责各自业务计算
- 月结中心负责统一汇总、对账、审批、锁定

### 2. 目标比例不是直接硬约束

本设计不推荐将“人员成本 30%、跟投 20%、公司留存 50%”做成硬性强制分摊规则。

推荐模式为：

- 目标比例：经营计划值
- 实际金额：系统汇总的真实发生值
- 差异审批：当目标与实际不一致时，要求说明、调整或退回上游核对

原因：

- 月末奖金、社保、补贴、跟投审核等因素会造成实际金额波动
- 若比例作为硬约束，会让月结长期处于“算不平”的状态
- 经营上仍需要比例抓手，因此不能完全放弃比例

### 3. 人员提成属于人员成本，不是独立大类

公司月结维度下，店铺纯利润按店铺绑定人员和提成比例拆分得到的金额，本质上属于“人员成本”的子项，而不是与人员成本并列的一条主线。

因此月结总账中的主类固定为：

- 人员成本
- 跟投收益
- 公司留存
- 调整项

其中人员成本再细分为：

- 店铺提成
- 固定工资
- 绩效工资
- 临时奖金
- 社保公积金
- 其他人工成本

### 4. 月结中心放在流程末端

月结中心不是先拍比例、再让上下游配合的入口。

推荐顺序是：

- 上游利润、提成、绩效、工资、跟投先基本稳定
- 再进入月结中心统一汇总、核对、审批、锁定

## Current System Mapping

### 已有能力

#### 1. 店铺月度结算基准利润

现有 `profit_basis_amount` 已可作为店铺正式结算利润口径，当前公式为：

```text
profit_basis_amount = orders_profit_amount - a_class_cost_amount
```

这部分继续保留，作为月结中心的核心上游输入。

#### 2. 店铺人员归属与提成比例

现有“人员店铺归属和提成比”模块负责：

- 店铺绑定主管/运营
- 配置人员提成比例
- 配置店铺可分配净利润率

该模块未来继续存在，但职责从“局部结算页面”收敛为“店铺提成来源模块”。

#### 3. 员工工资单

现有工资单闭环已经基本建立：

- 固定薪资来自 `salary_structures`
- 月度录入来自工资单手工字段
- 系统自动汇总 `gross_salary / net_salary / total_cost`

该模块未来继续作为“人员成本来源模块”。

#### 4. 跟投收益

现有跟投收益逻辑已经可以按店铺、按月基于 `profit_basis_amount` 试算、审批。

该模块未来继续作为“跟投收益来源模块”。

### 当前缺口

当前仍缺：

- 公司级月度利润结算主表
- 公司级月度利润结算明细表
- 统一月结接口
- 统一月结页面
- 统一审批与锁定机制
- 目标比例与实际金额对比机制

## Target Business Model

### 公司月度总公式

月结中心采用以下统一总公式：

```text
公司月度净利润 = 人员成本总额 + 跟投收益总额 + 公司留存总额 + 调整项
```

等价地：

```text
公司留存总额 = 公司月度净利润 - 人员成本总额 - 跟投收益总额 - 调整项
```

### 人员成本结构

人员成本总额由以下项目构成：

- 店铺提成
- 固定工资
- 绩效工资
- 临时奖金
- 社保公积金
- 加班费
- 补贴
- 其他人工成本

### 目标比例结构

经营上允许按月设置目标比例：

- 人员成本目标比例
- 跟投收益目标比例
- 公司留存目标比例

目标比例需要满足：

```text
personnel_target_ratio + follow_target_ratio + company_target_ratio = 100%
```

### 实际结算结构

系统同时计算实际金额和实际占比：

- `personnel_actual_amount`
- `follow_actual_amount`
- `company_actual_amount`
- `personnel_actual_ratio`
- `follow_actual_ratio`
- `company_actual_ratio`

### 差异机制

系统展示：

- 目标金额
- 实际金额
- 差异金额
- 差异比例

当差异超出阈值时，不允许直接锁定，必须选择一种处理方式：

- 调整目标比例
- 录入月结调整项
- 退回上游模块核对数据
- 填写差异说明并提交审批

## Module Responsibilities After Refactor

### 1. 人员店铺归属和提成比模块

未来职责：

- 维护店铺与人员绑定关系
- 维护人员提成比例
- 维护店铺可分配净利润率
- 生成“店铺提成”来源数据

不再负责：

- 公司级月度人员成本总账
- 公司级月度净利润拆分

月结中的角色：

- 作为“人员成本 > 店铺提成”子项来源

### 2. 员工薪资模块

未来职责：

- 维护固定薪资结构
- 维护绩效比例
- 维护默认提成比例
- 维护社保、公积金基数
- 维护奖金、个税、扣款、公司承担社保等月度字段
- 生成工资单与 `total_cost`

不再负责：

- 独立定义公司月度利润分配策略

月结中的角色：

- 作为“人员成本 > 工资相关成本”来源

### 3. 跟投收益模块

未来职责：

- 继续维护跟投记录
- 继续计算、审批、回退店铺级跟投结算
- 对外提供月度跟投实际金额

月结中的角色：

- 作为“跟投收益”来源

### 4. 月度利润结算中心

新增职责：

- 汇总公司月度净利润
- 汇总人员成本
- 汇总跟投收益
- 自动反推公司留存
- 展示目标比例与实际金额差异
- 统一审批与锁定

## Recommended Monthly Operation Flow

以下流程适用于例如在 `2026-05-10` 处理 `2026-04` 的月结。

### Step 1. 确认利润基础数据齐全

先确认 4 月利润基础数据已经稳定，包括：

- 订单利润
- 广告等 A 类费用
- `profit_basis_amount`

若店铺净利润口径尚未稳定，不应进入月结中心。

### Step 2. 确认店铺归属与提成配置

在“人员店铺归属和提成比”模块中确认：

- 当月店铺归属关系
- 提成比例
- 可分配净利润率

这一步会直接影响“店铺提成”金额。

### Step 3. 重算绩效

确认绩效计算已完成，并得到最新绩效结果。

当前系统已支持“绩效重算 -> 个人提成 -> 个人绩效 -> 工资单生成”的主链路，因此这一步应在工资单确认之前完成。

### Step 4. 确认员工工资与月度录入字段

在工资单或薪资模块中确认：

- 固定薪资结构
- 绩效工资来源
- 奖金
- 加班费
- 个人社保
- 个人公积金
- 个税
- 公司承担社保、公积金
- 其他扣款或补贴

这一步完成后，可得到人员成本中的大部分工资类金额。

### Step 5. 确认跟投收益

确认当月所有店铺跟投收益试算与审批结果，得到：

- `follow_actual_amount`

### Step 6. 进入月度利润结算中心

在月结中心中完成：

- 设置当月目标比例
- 自动读取实际金额
- 查看目标与实际差异
- 录入必要调整项

### Step 7. 差异处理

若差异超阈值，则执行：

- 调整目标比例
- 或回退上游核对
- 或录入差异说明后提审

### Step 8. 审批并锁定月份

当总账平衡且审批通过后，锁定该月月结数据。

锁定后：

- 月结中心不再允许直接改写金额
- 上游如需更改，必须先 reopen 月结，再重跑流程

## Performance Coefficient Treatment

### Current Status

当前仓库中的“绩效系数”口径并不统一：

- 员工侧公开列表中存在线性换算逻辑
- 店铺绩效写库时当前又可能写入固定值

因此，当前系统尚不能视为已经正式落地某一套稳定的“排名倍率规则”。

### Recommendation

绩效系数不要在本轮月结改造中继续保持模糊状态，而应作为单独治理项处理。

建议将其拆成独立规则：

- 先明确绩效系数到底是“按排名离散分档”还是“按分数线性换算”
- 再明确它影响哪一部分：
  - 店铺提成
  - 绩效工资
  - 两者都影响
  - 仅展示不参与结算

### Recommended First-Phase Boundary

第一阶段月结改造中：

- 月结中心只读取绩效结果带来的最终工资单/提成结果
- 不在月结中心重新计算绩效系数
- 不在本设计中强行确定“1.2 / 1.1 / 1.0 / 0.9 / 0.8”规则

后续可单独补一份“绩效系数治理设计”，明确：

- 系数规则
- 适用对象
- 生效时机
- 历史月份追溯规则

## API Proposal

### 1. `GET /api/finance/monthly-profit-settlement`

用途：

- 查询某个月的公司级月结结果

返回内容建议包括：

- 主表摘要
- 人员成本汇总
- 跟投收益汇总
- 公司留存
- 差异信息

### 2. `POST /api/finance/monthly-profit-settlement/rebuild`

用途：

- 按月份从上游模块重建月结草稿

行为：

- 重新汇总净利润
- 重新汇总人员成本
- 重新汇总跟投收益
- 生成或更新月结草稿

### 3. `PUT /api/finance/monthly-profit-settlement/{id}/targets`

用途：

- 更新目标比例与目标金额

### 4. `POST /api/finance/monthly-profit-settlement/{id}/approve`

用途：

- 审批月结

### 5. `POST /api/finance/monthly-profit-settlement/{id}/reopen`

用途：

- 回退已审批月结，使其重新进入草稿状态

## Workflow Impact

### 对现有系统的影响

影响程度为中等，可控。

### 不变的部分

- `profit_basis_amount` 店铺利润基准继续保留
- 人员店铺归属和提成比页面继续保留
- 员工薪资与工资单继续保留
- 跟投收益模块继续保留

### 会变化的部分

- 新增公司级月结入口
- 月末流程从“分散模块分别看结果”变为“统一月结中心汇总审批”
- 上游模块需要支持月结锁定后的回退与重算协同

### 流程变化总结

从：

```text
各模块各自结算
```

变为：

```text
上游模块出数 -> 月结中心汇总 -> 差异处理 -> 审批锁定
```

## Rollout Strategy

### Phase 1

- 新增月结主表
- 新增汇总接口
- 新增月结页面
- 只汇总：
  - `profit_basis_amount`
  - 店铺提成
  - 工资单人工成本
  - 跟投收益

### Phase 2

- 增加差异阈值与审批提示
- 增加调整项录入
- 增加审批与回退日志

### Phase 3

- 若业务需要，再增加人工成本按店铺分摊能力
- 若业务需要，再接入预算与实际对比分析
- 单独治理绩效系数规则

## Open Decisions

以下问题需要业务确认后进入实现计划：

1. 月结锁定后的 reopen 权限由谁持有？
2. 差异阈值的默认规则是什么？按金额、按比例，还是两者同时判断？
3. 固定工资与社保是否在第一阶段只做公司级汇总，不回摊店铺？
4. 跟投收益以“approved”还是“paid”作为月结中心的正式实际值？
5. 绩效系数最终是否参与提成、工资，还是仅作展示？

## Recommendation Summary

本设计的最终推荐是：

- 保留现有 HR、薪资、跟投、店铺提成模块
- 新增“月度利润结算中心”作为公司级总账
- 使用“目标比例 + 实际结算 + 差异审批”模式
- 将店铺提成明确归类为人员成本
- 将绩效系数治理独立成后续专项，而不是在本轮月结改造中混入

这是与现有系统衔接成本最低、业务解释最清楚、后续扩展空间最大的方案。
