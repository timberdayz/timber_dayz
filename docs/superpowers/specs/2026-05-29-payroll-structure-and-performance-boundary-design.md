# 工资结构与绩效边界重构设计

## 目标

将现有“店铺绩效 / 个人绩效 / 提成 / 工资单”链路重构为职责清晰、口径稳定、符合跨境电商业界习惯的结构，重点解决以下问题：

1. 底薪不应参与绩效乘法。
2. 店铺绩效与个人绩效边界不清。
3. 运营目标同时承担店铺与个人语义，导致配置复杂且结果难解释。
4. “我的收入”结果可见，但来源解释不清。

## 最终工资口径

采用以下正式口径：

```text
固定薪资 = 底薪 + 固定岗位工资 + 固定补贴
绩效工资 = 独立绩效包 × 个人绩效系数
提成 = 店铺利润基数 × 可分配利润率 × 个人提成比例 × 店铺绩效系数
工资单 = 固定薪资 + 绩效工资 + 提成 + 奖金/补贴 - 扣款
```

## 为什么要这样改

### 1. 底薪必须固定

底薪是员工的保底收入，不应成为绩效浮动基座。若把底薪与岗位工资一起乘 `performance_ratio × performance_score`，会导致：

- 绩效工资规模被底薪放大
- 固定收入与浮动收入边界模糊
- 员工难以理解“绩效工资到底从哪来”

### 2. 店铺绩效与个人绩效的职责不同

当前系统更接近：

- 店铺绩效：自动计算较完整
- 个人绩效：继承店铺分数，再叠加考勤/人工调整

这会导致：

- “李四做得怎么样”无法独立回答
- 店铺目标和个人目标混淆
- 工资解释必须倒推多层链路

## 重构后的四层结构

### A. 店铺绩效层

职责：

- 只回答“这个店本月经营得怎么样”
- 输出店铺绩效分和店铺绩效系数

输入：

- 店铺销售目标
- 店铺利润目标
- 店铺重点商品目标
- 店铺运营目标

输出：

- `shop_performance_score`
- `performance_coefficient`

说明：

- 店铺绩效系数只作用于提成
- 不直接决定个人绩效工资

### B. 个人绩效层

职责：

- 只回答“李四本月个人做得怎么样”
- 输出个人绩效得分 / 个人绩效系数

输入：

- 个人销售目标
- 个人运营任务
- 考勤
- 人工奖惩

输出：

- `employee_performance_score`
- `employee_performance_coefficient`

说明：

- 不再直接继承店铺总分作为个人分
- 如需保留“挂店承接责任”，也应作为个人绩效中的一个权重项，而不是唯一来源

### C. 提成计算层

职责：

- 计算员工因负责店铺而获得的提成

输入：

- `shop_profit_basis.profit_basis_amount`
- `shop_commission_config.allocatable_profit_rate`
- `employee_shop_assignments.commission_ratio`
- 店铺绩效系数

输出：

- `employee_commissions.commission_amount`

说明：

- 提成仍然与店铺经营结果绑定
- 继续由店铺绩效系数调节

### D. 工资单层

职责：

- 汇总员工最终工资

输入：

- 固定薪资
- 绩效工资
- 提成
- 奖金/补贴
- 扣款

输出：

- `payroll_records`

说明：

- 工资单是“我的收入”和月结人员成本最终真源
- 工资单不再承担解释逻辑，只承担汇总结果

## 运营目标重构原则

### 现状问题

当前“运营目标”配置项包含：

- 指标
- 方向
- 目标值 / 实际值
- 满分
- 罚分开关 / 阈值 / 每单位罚分 / 最大罚分
- 人工评分开关 / 人工打分值

最终却主要进入店铺 `operation_score`。这导致：

- 配置复杂
- 语义不清
- 用户不知道这是店铺目标还是个人目标

### 目标结构

运营目标必须拆成两类：

1. 店铺运营目标
   - 进入店铺绩效
   - 影响 `shop_performance_score.operation_score`

2. 个人运营目标
   - 进入个人绩效
   - 影响 `employee_performance_score`

禁止继续使用一套“运营目标”同时服务两层绩效。

## 建议的数据模型方向

### 固定薪资层

保留并重构现有薪资结构，使其至少能表达：

- `base_salary`
- `fixed_position_salary`
- `fixed_allowances`
- `performance_package_amount`
- `performance_ratio`（可选，若未来仍要保留比例表达）
- `commission_ratio_default`

建议：

- `performance_package_amount` 成为绩效工资的正式基座
- 若存在 `performance_ratio`，也应只作用于绩效包，不作用于底薪

### 个人绩效层

建议让 `employee_performance` 扩展为真正的个人绩效结果表，至少保留：

- `employee_code`
- `year_month`
- `sales_score`
- `operation_score`
- `attendance_score`
- `manual_adjustment_score`
- `performance_score`
- `performance_coefficient`

### 店铺绩效层

继续保留 `performance_scores`，但语义明确为店铺级：

- `sales_score`
- `profit_score`
- `key_product_score`
- `operation_score`
- `total_score`
- `performance_coefficient`

## 工资单重构建议

### 应发项

- `fixed_salary_total`
  - 底薪 + 固定岗位工资 + 固定补贴
- `performance_salary`
  - 独立绩效包 × 个人绩效系数
- `commission`
  - 提成
- `bonus`
- `overtime_pay`

### 扣除项

- 个人社保
- 个人公积金
- 个税
- 其他扣款

### 公司成本

- 公司社保
- 公司公积金

## 对李四链路的解释改进

按重构后设计，李四 2026-03 的解释应能直接回答：

1. 他负责哪些店？
2. 这些店的店铺绩效如何？
3. 他个人绩效如何？
4. 提成是多少，为什么？
5. 绩效工资是多少，为什么？
6. 最终工资单是多少？

这 6 个问题应分别由：

- 店铺归属模块
- 店铺绩效模块
- 个人绩效模块
- 提成结果
- 工资单

逐层直接回答，而不是让用户倒推。

## 页面职责重构建议

### 人员店铺归属和提成比

只做：

- 店铺归属
- 提成比例
- 可分配利润率

不再承担任何个人绩效含义。

### 绩效管理

拆为两个明确区域：

1. 店铺绩效
2. 个人绩效

运营目标入口也按这两类拆分。

### 我的收入

继续只读工资单口径，但说明层明确展示：

- 提成来源：店铺利润分配结果，含店铺绩效系数
- 绩效工资来源：独立绩效包 × 个人绩效系数
- 个人绩效来源：个人目标 + 考勤 + 奖惩

## 实施原则

因为当前仍在测试阶段，建议一步到位重构，不做长期兼容负担：

1. 先统一正式工资公式
2. 再拆店铺绩效与个人绩效模型
3. 再拆运营目标配置入口
4. 再改工资单生成逻辑
5. 最后同步“我的收入”和月结读取逻辑

## 结论

最终推荐：

- 店铺绩效只管店
- 个人绩效只管人
- 提成只吃店铺绩效系数
- 绩效工资只吃个人绩效系数
- 底薪绝不进入绩效乘法基座

这是当前仓库从“结果可算”走向“结构合理、可解释、可维护”的必要重构方向。
