# Change: 网页登录验证码统一优化

## Why

1. **图形验证码导致登录卡死**：部分平台（如妙手 ERP）登录页新增或间歇性出现「请输入验证码」图形验证码，采集脚本在点击「立即登录」后无法识别该环节，长时间轮询仍停留在登录页，最终报「登录仍停留在登录页」并失败。
2. **三种验证码场景未统一**：当前存在三种需覆盖的场景——(A) 仅短信/电话验证码，需用户提供验证码后由采集系统代为输入；(B) 因登录错误次数触发的图形验证码；(C) 登录页强制图形验证码。TikTok/Shopee 在旧 CLI 已支持 (A)，新前端与执行器尚未统一「暂停 → 用户输入 → 回传 → 继续」的适配；(B)(C) 在新采集链路中缺少检测与处理。
3. **与 Playwright/业界实践对齐**：官方建议不尝试「破解」验证码，而是检测验证码出现、由人工或合规的第三方打码服务提供答案、脚本仅负责填入并继续。需在架构上统一「验证码检测 → 暂停/打码 → 填入 → 继续」的流程。

## What Changes

- **统一验证码类型与执行模型**：将登录过程中的验证码抽象为 (1) 短信/电话验证码（OTP）、(2) 图形验证码（含偶发与必出）。执行器侧仅负责**检测**（如存在「请输入验证码」+ 验证码输入框/图片）与**执行**（从上下文或打码服务取得答案后填入、点击提交）；答案来源可插拔：**人工回填**（复用现有 VerificationRequired + paused + 前端输入）或**可选第三方打码 API**。
- **登录组件内增加验证码步骤**：在「点击登录」前或短轮询后（如 2–3 秒）检测是否出现图形验证码区域；若存在则进入「验证码处理」分支：未配置打码且上下文中无用户回传的验证码时，抛出 `VerificationRequired(verification_type="graphical_captcha", screenshot_path=...)`，由执行器将任务置为 paused 并返回截图/输入提示；若已有验证码（用户回传或打码结果），则填入并点击登录后继续判断是否成功。
- **妙手登录组件改造**：在 `modules/platforms/miaoshou/components/login.py` 中，在点击「立即登录」之后若仍停留在登录页，检测「请输入验证码」及图形验证码输入框/图片；若存在则按上述统一流程处理（暂停等待用户输入或调用打码服务），避免无限等待 15 秒后失败。
- **前端与 API 对接**：与现有 `verification_required`、`verification_type`、`screenshot_path` 及任务 paused 状态对接；前端在任务暂停且类型为图形验证码时展示截图与验证码输入框，用户提交后通过既有或新增 API 将验证码回传，后端写入任务/会话上下文并恢复任务或重试登录步骤。
- **可选：第三方打码服务适配**：若配置了 2Captcha/Anti-Captcha 等，在检测到图形验证码时可选地截取验证码图片并调用打码 API，将返回结果填入；仅在配置开启且合规前提下使用，人工回填作为默认或必选路径。

## 验证码场景与处理一览

当前定义**四类**场景（三类需验证码、一类无验证码），抽象为 **(1) OTP** 与 **(2) 图形验证码** 两类统一处理；详见本 change 下 `验证码场景与处理一览.md`。

| 场景 | 类型 | 处理要点 |
|------|------|----------|
| 1. 短信/电话验证码（OTP） | OTP | 检测 → VerificationRequired(otp) → paused → 前端输入 → 写入 config['otp'] → 恢复 → 从 config 填入并提交 |
| 2. 图形验证码（偶发/触发） | 图形 | 点击登录后短等 2–3 秒，仍停留登录页且存在验证码 DOM/文案 → 无 config 则截图并 VerificationRequired(graphical_captcha) → 人工回填或可选打码 → 写入 config['captcha_code'] → 填入并点击登录 |
| 3. 图形验证码（始终出现） | 图形 | 与场景 2 同一套检测与回填；可在填账号密码前先查一次 |
| 4. 无验证码 | — | 不进入验证码分支，按现有成功/失败逻辑处理，不做长时间傻等 |

**统一约定**：检测与回填有头/无头同一套；答案仅来自人工回填或（可选）合规第三方打码 API，不本地破解。

## 检测与应对设计（有头/无头一致）

### 检测方式（业界主流）

- **DOM/选择器（主手段）**：用 Playwright 检查页面上是否出现验证码相关元素。典型包括——验证码输入框（如 `input[name="captcha"]`、`#captcha`、`input[placeholder*="验证码"]`）、验证码图片（如 `img.captcha-img`、`img[src*="captcha"]` 或验证码 iframe）、以及包含「请输入验证码」「验证码错误」等文案的节点。判断条件为元素**可见**（`locator.is_visible()`）或存在（`locator.count() > 0`），必要时在短时内（如 2–3 秒）`wait_for` 出现。选择器可按平台/站点配置，便于多站点维护。
- **检测时机**：**点击登录之后**——点击「立即登录」→ 短等待 2–3 秒 → 若仍停留在登录页且未出现成功跳转特征，再检查上述验证码 DOM/文案；一旦判定「需要验证码」即进入回填流程，**不做长时间轮询或盲目重试**。若目标站是「一打开登录页就出验证码」，可在填账号密码前先查一次，有则先走验证码分支再点登录。
- **URL/停留页与文案（辅助）**：用「当前是否仍为登录 URL」或「是否未出现登录成功后才有的元素」辅助判断；结合文案区分图形验证码与 OTP（如「短信验证码」），便于走不同回填路径。检测逻辑**有头与无头完全相同**，仅依赖 DOM、URL、文案，与浏览器是否可见无关。

### 回填方式（业界主流）

- **人工回填（主路径）**： 1) 检测到验证码后，脚本截全屏或验证码区域（`page.screenshot()`），保存路径，并抛出/返回「需要验证码」状态（如 `VerificationRequired`，带 `verification_type`、`screenshot_path`、`task_id`）。 2) 任务置为 `paused`，前端展示截图与验证码输入框，用户输入后提交。 3) 后端将验证码写入该任务执行上下文（如 `config['captcha_code']` / `config['otp']`）。 4) 恢复执行：执行器从 config 读取验证码，登录组件用 `page.fill(selector, value)` 填入验证码输入框，再 `click` 登录/提交按钮。 **实现形态**：同进程内「暂停等待回传后继续」时，用同一 `page` 直接 fill + click；若为跨进程/长时间暂停，可保存 `storage_state`，恢复时带验证码重新打开登录页并填账号+密码+验证码后点击登录，人工回填仍通过「提交验证码 API → 写入 config → 重试/恢复」完成。
- **第三方打码（可选）**：检测到图形验证码后，截取验证码区域或拿到站点 token，调用打码 API（如 2Captcha），轮询取回识别结果，用 `page.fill` 填入并点击提交；失败或未配置时 fallback 人工回填。人工回填为默认或必选能力，打码为可选增强。
- **回填细节**：回填使用的输入框选择器与检测时一致（或来自同一配置）；先 `fill` 验证码再 `click` 提交；人工回填可设「等待回传」超时（如 5 分钟）；验证码经 HTTPS 提交、仅写回该任务上下文、避免落日志，保证安全。

### 有头/无头与业界对齐

- 检测逻辑有头/无头**同一套**；应对流程按**无头生产**设计（必须走「暂停 → 回传 → 填入」），有头测试可走同一套以保证与生产一致。不尝试本地破解验证码；与 Playwright 官方及企业采集主流做法一致——人工回填为主、打码服务为可选。

### 前端发现与入口：轮询为主、WebSocket 为辅

- **主路径（轮询）**：用户通过**轮询**任务列表或任务详情发现任务状态为 `paused`；在采集任务页（如 `/collection-tasks`）该任务行点击「继续」，打开验证码弹窗（展示截图与输入框），输入后提交并调用恢复 API（如 `resumeTask`）。不依赖 WebSocket 长连接；与现有多数功能「轮询」设计一致，且便于**云端/多设备**场景——用户可在任意设备打开系统，看到暂停任务并完成验证。
- **后端持久化**：执行器将任务置为 `paused` 并抛出 `VerificationRequired` 时，须将 `verification_type`、`screenshot_path`（或可访问的截图 URL）**持久化到任务记录**（如任务表或任务详情接口返回字段），以便前端仅凭轮询即可拿到截图并展示，无需依赖实时推送。
- **WebSocket 为可选增强**：若前端已为该任务建立 WebSocket，可在此基础上在收到 `verification_required` 时**自动弹出**验证码弹窗，减少一次手动点「继续」；此为同页面的体验优化，**非主路径**。系统必须在不依赖 WebSocket 的前提下完成「发现暂停 → 展示截图 → 人工回填 → 恢复」全流程。

## Impact

- **Affected specs**: data-collection（ADDED：登录验证码处理相关需求与场景）
- **Affected code**:
  - `modules/platforms/miaoshou/components/login.py`：增加图形验证码检测与处理分支
  - `modules/apps/collection_center/executor_v2.py`：复用 VerificationRequired/verification_required/verification_type/screenshot_path，支持 graphical_captcha 类型及恢复时注入验证码
  - 各平台登录组件（TikTok/Shopee 等）：统一从 `ctx.config` 或任务上下文读取 OTP/图形验证码，与现有 OTP 逻辑对齐
  - 前端：任务详情/采集页在任务 paused 且 verification_type 为图形验证码时展示截图与输入框，提交后调用回传 API
  - 可选：新增打码服务适配模块（如 `modules/apps/collection_center/captcha_solver.py`）及配置项

## Non-Goals

- 不实现验证码「破解」或本地识别模型；答案仅来自人工输入或合规的第三方打码 API。
- 不强制所有平台接入打码服务；人工回填为默认能力，打码为可选增强。
- 测试环境若可 stub 或关闭验证码，本变更不强制要求；仅在生产/真实采集场景下保证「检测 → 暂停或打码 → 填入 → 继续」可用。
