# Change: 网页登录验证码统一优化

## Why

1. **图形验证码导致登录卡死**：部分平台（如妙手 ERP）登录页新增或间歇性出现「请输入验证码」图形验证码，采集脚本在点击「立即登录」后无法识别该环节，长时间轮询仍停留在登录页，最终报「登录仍停留在登录页」并失败。
2. **三种验证码场景未统一**：当前存在三种需覆盖的场景——(A) 仅短信/电话验证码，需用户提供验证码后由采集系统代为输入；(B) 因登录错误次数触发的图形验证码；(C) 登录页强制图形验证码。TikTok/Shopee 在旧 CLI 已支持 (A)，新前端与执行器尚未统一「暂停 → 用户输入 → 回传 → 继续」的适配；(B)(C) 在新采集链路中缺少检测与处理。
3. **与 Playwright/业界实践对齐**：官方建议不尝试「破解」验证码，而是检测验证码出现、由人工或合规的第三方打码服务提供答案、脚本仅负责填入并继续。需在架构上统一「验证码检测 → 暂停/打码 → 填入 → 继续」的流程。

## What Changes

- **统一验证码类型与执行模型**：将登录过程中的验证码抽象为 (1) 短信/电话验证码（OTP）、(2) 图形验证码（含偶发与必出）。执行器侧仅负责**检测**（如存在「请输入验证码」+ 验证码输入框/图片）与**执行**（从上下文或打码服务取得答案后填入、点击提交）；答案来源可插拔：**人工回填**（复用现有 VerificationRequiredError + paused + 前端输入）或**可选第三方打码 API**。
- **登录组件内增加验证码步骤**：在「点击登录」前或短轮询后（如 2–3 秒）检测是否出现图形验证码区域；若存在则进入「验证码处理」分支：未配置打码且上下文中无用户回传的验证码时，抛出 `VerificationRequiredError(verification_type="graphical_captcha", screenshot_path=...)`，由执行器将任务置为 paused 并返回截图/输入提示；若已有验证码（用户回传或打码结果），则填入并点击登录后继续判断是否成功。
- **妙手登录组件改造**：在 `modules/platforms/miaoshou/components/login.py` 中，在点击「立即登录」之后若仍停留在登录页，检测「请输入验证码」及图形验证码输入框/图片；若存在则按上述统一流程处理（暂停等待用户输入或调用打码服务），避免无限等待 15 秒后失败。
- **前端与 API 对接**：与现有 `verification_required`、`verification_type`、`screenshot_path` 及任务 paused 状态对接；前端在任务暂停且类型为图形验证码时展示截图与验证码输入框，用户提交后通过既有或新增 API 将验证码回传，后端写入任务/会话上下文并恢复任务或重试登录步骤。
- **可选：第三方打码服务适配**：若配置了 2Captcha/Anti-Captcha 等，在检测到图形验证码时可选地截取验证码图片并调用打码 API，将返回结果填入；仅在配置开启且合规前提下使用，人工回填作为默认或必选路径。

## 验证码场景与处理一览

当前定义**四类**场景（三类需验证码、一类无验证码），抽象为 **(1) OTP** 与 **(2) 图形验证码** 两类统一处理；详见本 change 下 `验证码场景与处理一览.md`。

| 场景 | 类型 | 处理要点 |
|------|------|----------|
| 1. 短信/电话验证码（OTP） | OTP | 检测 → VerificationRequiredError(otp) → paused → 前端输入 → 写入存储（实现上 Redis）→ 执行器在同一 page 从存储读取并填入、提交 |
| 2. 图形验证码（偶发/触发） | 图形 | 点击登录后短等 2–3 秒，仍停留登录页且存在验证码 DOM/文案 → 无回传则截图并 VerificationRequiredError(graphical_captcha) → 人工回填或可选打码 → 写入存储（实现上 Redis）→ 执行器在同一 page 从存储读取并填入、点击登录 |
| 3. 图形验证码（始终出现） | 图形 | 与场景 2 同一套检测与回填；可在填账号密码前先查一次 |
| 4. 无验证码 | — | 不进入验证码分支，按现有成功/失败逻辑处理，不做长时间傻等 |

**统一约定**：检测与回填有头/无头同一套；答案仅来自人工回填或（可选）合规第三方打码 API，不本地破解。

## 检测与应对设计（有头/无头一致）

### 检测方式（业界主流）

- **DOM/选择器（主手段）**：用 Playwright 检查页面上是否出现验证码相关元素。典型包括——验证码输入框（如 `input[name="captcha"]`、`#captcha`、`input[placeholder*="验证码"]`）、验证码图片（如 `img.captcha-img`、`img[src*="captcha"]` 或验证码 iframe）、以及包含「请输入验证码」「验证码错误」等文案的节点。判断条件为元素**可见**（`locator.is_visible()`）或存在（`locator.count() > 0`），必要时在短时内（如 2–3 秒）`wait_for` 出现。选择器可按平台/站点配置，便于多站点维护。
- **检测时机**：**点击登录之后**——点击「立即登录」→ 短等待 2–3 秒 → 若仍停留在登录页且未出现成功跳转特征，再检查上述验证码 DOM/文案；一旦判定「需要验证码」即进入回填流程，**不做长时间轮询或盲目重试**。若目标站是「一打开登录页就出验证码」，可在填账号密码前先查一次，有则先走验证码分支再点登录。
- **URL/停留页与文案（辅助）**：用「当前是否仍为登录 URL」或「是否未出现登录成功后才有的元素」辅助判断；结合文案区分图形验证码与 OTP（如「短信验证码」），便于走不同回填路径。检测逻辑**有头与无头完全相同**，仅依赖 DOM、URL、文案，与浏览器是否可见无关。

### 回填方式（业界主流）

- **人工回填（主路径）**： 1) 检测到验证码后，脚本截全屏或验证码区域（`page.screenshot()`），保存路径，并抛出/返回「需要验证码」状态（如 `VerificationRequiredError`，带 `verification_type`、`screenshot_path`、`task_id`）。 2) 任务置为 `paused`，前端展示截图与验证码输入框，用户输入后提交。 3) 后端将验证码写入约定存储（如 Redis keyed by task_id）。 4) 恢复执行：**同一进程、同一 page** 上，执行器从存储读取验证码，用 `page.fill(selector, value)` 填入验证码输入框，再 `click` 登录/提交按钮，然后继续后续流程。 **实现形态（业界主流）**：**同一进程、同一页面内等待回传后继续**——检测到验证码时任务置为 paused 并持久化截图/类型，但**执行进程与浏览器/page 不退出**，而是**阻塞等待**（如轮询 Redis/API 或 resume 通道）直至用户提交验证码/OTP 或超时；用户提交后，**当前同一 page** 上 fill + click，然后继续。OTP 与图形验证码均采用此形态，因现代化登录页刷新或重开会使当次验证上下文失效（新会话、新验证码、OTP 常与当次请求绑定）。
- **第三方打码（可选）**：检测到图形验证码后，截取验证码区域或拿到站点 token，调用打码 API（如 2Captcha），轮询取回识别结果，用 `page.fill` 填入并点击提交；失败或未配置时 fallback 人工回填。人工回填为默认或必选能力，打码为可选增强。
- **回填细节**：回填使用的输入框选择器与检测时一致（或来自同一配置）；先 `fill` 验证码再 `click` 提交；人工回填可设「等待回传」超时（如 5 分钟）；验证码经 HTTPS 提交、仅写回该任务上下文、避免落日志，保证安全。

### 有头/无头与业界对齐

- 检测逻辑有头/无头**同一套**；应对流程按**无头生产**设计（必须走「暂停 → 回传 → 填入」），有头测试可走同一套以保证与生产一致。不尝试本地破解验证码；与 Playwright 官方及企业采集主流做法一致——人工回填为主、打码服务为可选。
- **恢复形态与业界一致**：业界主流为「不刷新、不重开页面，在同一页等待人工/打码结果后填入并继续」；本方案与之一致，**OTP 与图形验证码均采用同一页、同一会话内恢复**（进程与 page 保持存活并等待回传），不采用「paused 后进程退出再重新启动任务」——因现代化页面刷新会导致当次验证失效。

### 前端发现与入口：轮询为主、WebSocket 为辅

- **主路径（轮询）**：用户通过**轮询**任务列表或任务详情发现任务状态为 `paused`；在采集任务页（如 `/collection-tasks`）该任务行点击「继续」，打开验证码弹窗（展示截图与输入框），输入后提交并调用恢复 API（如 `resumeTask`）。不依赖 WebSocket 长连接；与现有多数功能「轮询」设计一致，且便于**云端/多设备**场景——用户可在任意设备打开系统，看到暂停任务并完成验证。
- **后端持久化**：执行器在需要验证码时（捕获 `VerificationRequiredError` 后、进入阻塞等待前）须通过**回调**将 `verification_type`、截图路径**持久化到任务记录**（任务表字段：`verification_type`、`verification_screenshot`；任务状态置为 paused；任务详情/列表接口返回上述字段），以便前端仅凭轮询即可拿到截图并展示。执行器不在此处 return，而是进入等待；持久化由调用方或执行器提供的回调完成。
- **截图与任务接口**：获取验证码截图的 API（如 `GET /tasks/{task_id}/screenshot`）须读取任务表的 **`verification_screenshot`** 字段（与 schema 一致）；任务详情/列表接口须返回 `verification_type`、`verification_screenshot`，供前端展示截图与输入框。**多实例/分布式部署**：若 API 多实例或执行器与 API 非同机，截图须写入**共享存储**（如 NFS、对象存储），任务表存可被任意实例访问的路径或 URL；或约定验证码截图能力仅在单实例或「任务与实例绑定（sticky/亲和）」部署下保证可用。
- **进程与部署**：验证码等待期间**执行该任务的进程须保持存活**；滚动重启、扩缩容可能导致等待中的任务被终止，用户提交的验证码将无效。部署时需评估等待超时（如 5 分钟）与重启策略的兼容性，或约定等待期间避免重启执行进程。
- **WebSocket 为可选增强**：若前端已为该任务建立 WebSocket，可在此基础上在收到 `verification_required` 时**自动弹出**验证码弹窗，减少一次手动点「继续」；此为同页面的体验优化，**非主路径**。系统必须在不依赖 WebSocket 的前提下完成「发现暂停 → 展示截图 → 人工回填 → 恢复」全流程。

### 恢复链路与「提交验证码」API 契约

- **请求体**：恢复/提交验证码 API（如 `POST /tasks/{task_id}/resume`）须接受请求体，例如 `{ "captcha_code": "..." }`（图形验证码）或 `{ "otp": "..." }`（短信/电话验证码），与任务 ID 一起传入。
- **存储**：后端将用户提交的验证码写入与执行器约定的存储（如 **Redis keyed by `task_id`**），供**正在等待中的同一执行进程**读取，确保能在**当前同一 page** 上填入。
- **恢复执行**：用户提交后，后端将验证码写入存储；**正在阻塞等待的执行器**（同一进程、同一 page）从存储读取验证码，在**当前 page** 上 `page.fill()` + `page.click()` 提交，然后继续后续采集。仅把任务状态改为 `running` 而不写入存储或执行器未在同一 page 上填入，视为未完成恢复链路。**不得**采用「paused 后进程退出、再重新启动任务并打开新页面」作为主路径——因现代化页面刷新/重开会导致当次验证码/OTP 失效。
- **恢复形态约定（业界主流）**：采用**同一进程、同一页面内等待回传后继续**。检测到需要验证码/OTP 时：任务置为 paused，截图/类型持久化供前端展示，但**执行进程与浏览器/page 保持存活**，执行器**阻塞等待**（如轮询 Redis keyed by task_id，或通过 resume 回调）直至用户提交验证码/OTP 或**超时（如 5 分钟）**。用户提交后：后端将验证码写入约定存储，**当前同一执行上下文**在**同一 page** 上 fill + click，然后继续。**超时后**：执行器须**结束等待**、将任务置为失败或专用状态（如 `verification_timeout`）、**return** 并释放浏览器，以便进程可复用。**OTP 与图形验证码均按此形态**，不刷新、不重开页面。若 paused 发生在**导出阶段**（某域遇到验证码），同样在同一 page 等待回传后填入并继续该域导出。
- **Resume 与「已超时/已结束」**：若执行器已因超时结束等待并退出，用户再提交验证码时，后端应检测任务已非「等待验证」状态（如已失败、verification_timeout、已完成），**返回 4xx** 并提示「验证已超时或任务已结束，请重新发起采集」，避免静默写入存储却无人消费。

### 登录阶段验证码暂停契约

- **不吞掉 VerificationRequiredError**：执行器中调用登录组件的入口（如 `_execute_python_component`）在捕获异常时，**不得**将 `VerificationRequiredError` 当作普通 `Exception` 吞掉并仅返回失败；须** re-raise**，由上层统一处理。
- **捕获后持久化并阻塞等待**：在执行「登录组件」的代码块外，须有单独的 `except VerificationRequiredError`：捕获后通过回调将 `verification_type`、`screenshot_path` 持久化到任务表并将任务状态置为 paused（供前端轮询展示），**执行器不 return**，而是在**同一进程内阻塞等待**（如轮询 Redis keyed by task_id）直至用户提交验证码/OTP 或超时；收到后在**当前同一 page** 上 fill + click 并继续，最后再按正常流程 return 最终结果。从而登录阶段出现验证码时前端可展示截图与输入框，用户提交后同一 page 继续，无需重开页面。**双路径**：若仍保留 YAML/Playwright 登录路径，该路径同样需在捕获后持久化并阻塞等待，与 Python 路径一致。
- **导出阶段验证码**：在域循环内（导出某域时）若导出组件抛出 `VerificationRequiredError`，执行器同样**不得 return**；须在循环内捕获后通过回调持久化并阻塞等待，收到回传后在**同一 page** 上填入并继续该域导出，再继续后续域或结束，与登录阶段行为一致。

### 妙手登录实现约束

- 在「点击立即登录」之后：**缩短等待**（如 2–3 秒内判定），**增加图形验证码 DOM 检测**（如「请输入验证码」文案、验证码输入框/图片）；若检测到验证码则按统一流程抛出 `VerificationRequiredError` 并附截图，**避免**固定 15 秒轮询仍停留在登录页后才报错。

## Impact

- **Affected specs**: data-collection（ADDED：登录验证码处理相关需求与场景）
- **Affected code**:
  - `modules/platforms/miaoshou/components/login.py`：增加图形验证码检测与处理分支（2–3 秒内检测、避免 15 秒傻等）
  - `modules/apps/collection_center/executor_v2.py`：复用 VerificationRequiredError/verification_required/verification_type/screenshot_path，支持 graphical_captcha 类型；检测到验证码时**不退出进程**，而是将任务置为 paused、持久化截图/类型后**阻塞等待**（轮询 Redis 等）直至用户提交或**超时**，超时后结束等待、置任务失败或 verification_timeout 并 return 释放浏览器；收到回传后在**同一 page** 上 fill + click 并继续。**登录阶段**与**导出阶段（域循环内）**捕获 VerificationRequiredError 后均持久化并阻塞等待，不 return；且 `_execute_python_component` 不吞掉该异常
  - `backend/routers/collection.py`：Resume 接口在任务仍为「等待验证」状态时接收验证码请求体并写入约定存储（如 Redis keyed by task_id）；若任务已非等待验证（已超时/失败/完成）则返回 4xx 并提示重新发起；GET screenshot 使用 task.verification_screenshot；多实例时截图须来自共享存储或约定部署约束。执行器在验证码暂停时须保持进程与 page 存活并阻塞等待，而非退出进程
  - 任务详情/列表接口：返回 verification_type、verification_screenshot，供前端轮询展示。若采用专用状态 **verification_timeout**，须在任务状态枚举或 schema 中支持；否则可复用 `failed` 并通过 error_message 区分「验证超时」
  - 各平台登录组件（TikTok/Shopee 等）：统一从 `ctx.config` 或任务上下文读取 OTP/图形验证码，与现有 OTP 逻辑对齐
  - 前端：任务详情/采集页在任务 paused 且 verification_type 为图形验证码时展示截图与输入框，提交后调用回传 API
  - 可选：新增打码服务适配模块（如 `modules/apps/collection_center/captcha_solver.py`）及配置项

## Non-Goals

- 不实现验证码「破解」或本地识别模型；答案仅来自人工输入或合规的第三方打码 API。
- 不强制所有平台接入打码服务；人工回填为默认能力，打码为可选增强。
- 测试环境若可 stub 或关闭验证码，本变更不强制要求；仅在生产/真实采集场景下保证「检测 → 暂停或打码 → 填入 → 继续」可用。
