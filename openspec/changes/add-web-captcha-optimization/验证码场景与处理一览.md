# 验证码场景与处理一览

本文档梳理当前提案中定义的验证码场景及统一处理方式，便于实现与验收时对照。

---

## 一、验证码场景（按类型与触发条件）

| 序号 | 场景名称 | 类型 | 触发条件 | 说明 |
|------|----------|------|----------|------|
| 1 | **短信/电话验证码（OTP）** | OTP | 登录流程中平台要求输入手机/邮箱收到的验证码 | 如 TikTok、Shopee 等平台的二次验证；用户在前端输入后由系统代为填入 |
| 2 | **图形验证码（偶发/触发）** | 图形验证码 | 因多次登录错误、风控等触发的图形验证码 | 如妙手 ERP 点击「立即登录」后才出现「请输入验证码」；需检测到后再处理 |
| 3 | **图形验证码（始终出现）** | 图形验证码 | 登录页一打开即要求图形验证码 | 与场景 2 检测与处理流程相同，仅出现时机不同（可在填账号密码前先查一次） |
| 4 | **无验证码** | — | 点击登录后未出现验证码且页面已跳转或即将跳转 | 不进入验证码分支，按现有登录成功/失败逻辑处理，不做长时间傻等 |

**抽象层面**：系统将验证码抽象为两类——**(1) OTP**、**(2) 图形验证码**（偶发与必出共用同一套检测与回填）。场景 4 为「无需验证码」的常态路径。

---

## 二、各场景的处理方式

### 场景 1：短信/电话验证码（OTP）

| 步骤 | 处理 |
|------|------|
| 检测 | 登录组件/执行器检测到页面要求 OTP（如出现「短信验证码」「验证码已发送」等文案或 OTP 输入框） |
| 暂停 | 抛出 `VerificationRequiredError(verification_type="otp", ...)`，任务置为 `paused`，前端展示 OTP 输入框 |
| 回传 | 用户在前端输入 OTP 并提交 → 后端写入任务上下文 `config['otp']` |
| 继续 | 恢复任务，登录组件从 `config['otp']` 读取并填入页面，点击提交，继续完成登录 |

### 场景 2 与 3：图形验证码（偶发或始终出现）

| 步骤 | 处理 |
|------|------|
| 检测 | 点击登录后短等待（2–3 秒），若仍停留在登录页且存在验证码相关 DOM/文案（输入框、验证码图、「请输入验证码」等），则判定需要图形验证码 |
| 分支 A：有人工回传或打码结果 | 若 `config['captcha_code']` 已有值（用户已回传或打码服务已返回），则直接填入并点击登录，按现有逻辑判断是否成功 |
| 分支 B：需人工输入 | 截图并抛出 `VerificationRequiredError(verification_type="graphical_captcha", screenshot_path=...)`，任务置为 `paused`，前端展示截图与验证码输入框 |
| 回传 | 用户输入验证码并提交 → 后端写入 `config['captcha_code']` |
| 继续 | 恢复任务或重试登录步骤，登录组件从 `config['captcha_code']` 读取并填入，再次点击登录 |
| 可选：第三方打码 | 若已配置打码服务（如 2Captcha），检测到图形验证码时可截图上发、轮询结果后填入；失败或未配置时 fallback 人工回填。人工回填仍为默认/必选能力 |

### 场景 4：无验证码

- 点击登录后未检测到验证码区域且页面已跳转或即将跳转 → 不进入「验证码处理」分支，按现有成功/失败逻辑处理，**不在点击登录后长时间傻等**（如 15 秒）才报错。

---

## 三、统一约定（有头/无头一致）

- **检测**：有头与无头使用同一套逻辑（DOM/选择器 + URL/文案），与浏览器是否可见无关。
- **回填**：按无头生产设计——必须走「检测 → 暂停 → 前端回传 → 恢复 → 脚本 fill + click」；有头测试可走同一套，保证与生产一致。**实现上**：后端将用户提交的验证码写入 **Redis keyed by task_id**，执行器轮询读取后在同一 page 上填入并点击提交（任务上下文在实现上即该存储，非进程内 config）。
- **恢复形态（业界主流）**：用户提交验证码后为**同一进程、同一页面内「接着输入验证码继续」**——执行进程与浏览器/page **不退出**，检测到验证码时置为 paused 并**阻塞等待**用户回传（如轮询 Redis）；用户提交后在同一 page 上 fill + click 并继续。与业界主流一致；现代化页面刷新会导致当次验证失效，故**不采用**「重新启动任务、打开新页面」。
- **答案来源**：仅来自**人工回填**或（可选）**合规第三方打码 API**，不尝试本地破解验证码。

## 四、与 spec 场景的对应关系

| spec 中的 Scenario | 对应上文 |
|--------------------|----------|
| 短信/电话验证码（OTP）由用户提供后填入 | 场景 1 |
| 图形验证码出现时暂停并等待人工输入 | 场景 2/3，分支 B |
| 可选第三方打码服务填入图形验证码 | 场景 2/3，可选打码 |
| 无验证码时不做额外等待 | 场景 4 |
| 有头与无头共用同一套检测与回填 | 第三节「统一约定」 |
