## Context

西虹 ERP 经过 `refactor-rules-and-code-patterns` 提案后，已建立 Agent-first 开发治理底座（规则分层、后端代码模式、异常体系、测试 fixture、重构风控）。但审视发现，面向 50~100 人同时使用的现代化 ERP，还缺少"生产级治理"层。

当前技术栈：FastAPI + SQLAlchemy 2.0 + PostgreSQL 15+ + Vue 3 + Element Plus + Pinia + Redis + Celery。已有 CI/CD 流程（GitHub Actions tag 触发），已有 Docker 容器化部署。

## Goals / Non-Goals

**Goals:**

1. 建立关键 API 的性能 SLO 和容量基线，引入压测门禁
2. 建立数据库迁移预演、灰度发布和回滚策略
3. 定义 RPO/RTO 目标和恢复演练标准
4. 定义数据四级分级和敏感数据访问控制规则
5. 创建前端 Agent 开发模板，与后端 `CODE_PATTERNS.md` 对齐
6. 建立 ERP 关键域的业务正确性验收标准

**Non-Goals:**

1. 不实现完整的 APM/监控平台（使用现有日志和中间件，不引入 Prometheus/Grafana 等）
2. 不重写现有前端代码（仅提供模板，新页面强制使用，旧页面按需迁移）
3. 不修改后端业务逻辑（本提案仅定义规则和模板）
4. 不强制所有现有表添加数据分级标记（仅标记 restricted 级数据）

**依赖边界说明：**
- 本提案依赖 `refactor-rules-and-code-patterns` 先完成 L3 文档结构重构（核心 7 个主题文件 + README 索引）
- 本提案的 spec 可以先行评审，但实际文档落地不应与 V1 的目录重构并行实施

## Decisions

### D1: 性能 SLO 与容量基线

**决策**：定义三级 API 性能目标，记录在 `PRODUCTION_READINESS.md` 中。

| API 类别 | p95 延迟目标 | p99 延迟目标 | 示例 |
|----------|-------------|-------------|------|
| 关键读接口 | < 300ms | < 800ms | 用户列表、目标列表、绩效列表 |
| 关键写接口 | < 800ms | < 2s | 创建订单、提交绩效、更新目标 |
| 复杂报表/导出 | < 3s | < 8s | 收入汇总、财务报表、数据看板 |

**页面加载目标**：
- 首屏数据页（含 API 请求）：< 2s
- 筛选刷新：< 1.5s
- 表单提交反馈：< 1s

**容量基线**（50~100 并发用户）：
- 数据库连接池：`pool_size=20, max_overflow=30`
- Redis 连接数：< 50
- Celery worker：>= 4
- 任务队列积压阈值：> 100 pending 触发告警

**慢查询门禁**：
- 单次查询 > 1s 必须在 PR 中说明原因和优化计划
- 新增接口未定义性能预期，不允许合并

### D2: 压测门禁

**决策**：高风险变更上线前必须执行压测。

**工具选择**：`k6`（轻量、脚本化、CI 友好）。

**最小压测场景**：
- 登录
- 用户列表（分页查询）
- 目标/绩效/收入列表
- 创建/更新操作
- 采集任务触发

**压测标准**：
- 50 并发用户持续 2 分钟
- 错误率 < 1%
- p95 延迟符合 D1 定义
- 数据库连接不溢出

**触发条件**：
- 数据库 schema 变更
- 连接池/缓存配置变更
- 新增复杂查询/报表
- 关键服务重构

### D3: 发布与迁移治理

**决策**：建立三级发布策略。

**数据库迁移预演**：
1. 所有 Alembic 迁移先在 staging 执行
2. 高风险迁移（涉及数据变更/列删除/表重命名）必须：备份确认 + 回滚步骤 + 执行窗口 + 责任人
3. 迁移脚本 PR 必须附：幂等性说明、回滚说明、数据影响说明

**灰度发布规则**：
- 涉及 UI 变更或核心流程变更的功能，必须使用 feature flag
- feature flag 生命周期 <= 30 天，过期必须清理
- flag 命名规范：`ff_{domain}_{feature}_{date}`

**回滚策略分层**：

| 层级 | 触发条件 | 操作 |
|------|---------|------|
| 代码回滚 | 功能 bug、性能退化 | `git revert` + 重新部署 |
| 迁移回滚 | 数据库变更导致故障 | `alembic downgrade` + 维护窗口 |
| 数据补救 | 数据损坏/误删 | 从备份恢复 + 增量重放 |

### D4: 灾备与恢复

**决策**：定义 RPO/RTO 目标和演练要求。

| 指标 | 目标 |
|------|------|
| RPO（允许丢失的数据量） | <= 15 分钟 |
| RTO（恢复到可用的时间） | <= 2 小时 |

**备份策略**：
- PostgreSQL 全量备份：每日 1 次
- WAL 归档：持续增量
- Redis：AOF 持久化 + 每日 RDB 快照
- 文件存储（采集下载文件）：每日增量备份

**恢复演练**：
- 频率：每月 1 次
- 范围：至少覆盖 PostgreSQL 全量恢复 + 关键表验证
- 恢复后验证：运行 `scripts/verify_restore.py`，检查关键表行数、最新记录时间、引用完整性

### D5: 数据分级与敏感访问

**决策**：定义四级数据分类，建立敏感数据访问规则。

| 级别 | 定义 | 示例 | 访问要求 |
|------|------|------|---------|
| public | 可公开 | 平台名称、产品类目 | 无限制 |
| internal | 内部使用 | 订单统计、采集配置 | 需登录 |
| sensitive | 敏感 | 销售额、利润、客户信息 | 需 RBAC + 审计日志 |
| restricted | 高度敏感 | 薪资、绩效、身份证/手机号、密钥、Token | 需 RBAC + 审计日志 + 导出审批 |

**restricted 数据规则**：
- 查看：仅限 Admin 或数据所有者
- 导出：必须二次确认 + 审计日志记录
- 日志：禁止明文出现在 logger 输出中（必须脱敏）
- 审计保留：>= 180 天

### D6: 前端 Agent 模式

**决策**：创建 `docs/DEVELOPMENT_RULES/FRONTEND_CODE_PATTERNS.md`，为前端 Agent 提供等效于后端 `CODE_PATTERNS.md` 的标准模板。

**模板清单**：

| 模板 | 用途 | 关键约束 |
|------|------|---------|
| API 函数模板 | `frontend/src/api/*.js` | 统一 try/catch、统一参数格式 |
| Pinia Store 模板 | `frontend/src/stores/*.js` | state/getters/actions 分离、loading 状态管理 |
| 列表页模板 | `frontend/src/views/` | 统一查询/分页/筛选/loading 模式 |
| 表单页模板 | `frontend/src/views/` | 统一校验/提交/反馈模式 |
| 权限按钮模板 | `frontend/src/components/` | `hasPermission()` 函数 + `v-if` 模式（项目实际使用方式） |
| 局部 loading 模板 | - | 禁止全屏 loading、支持 showLoading 参数 |
| 后台刷新模板 | - | 非关键数据静默刷新、乐观更新 |

**强制规则**（纳入 `.cursorrules`）：
- 页面不能直接写 `axios.get/post`，必须通过 `api/*.js`
- store 不能在 `actions` 或 `$patch()` 之外修改 state（项目约定，非 Pinia 框架限制，目的是保障多 Agent 协作下的可追溯性）
- 列表页必须使用统一分页查询模式
- 新页面强制使用模板，旧页面修改时建议同步迁移

### D7: ERP 业务验收标准

**决策**：定义关键业务域的正确性规则和验收清单。

**金额精度规则**：
- CNY 金额：`Decimal`，保留 2 位小数，数据库使用 `Numeric(15, 2)`（与现有 schema.py 一致）
- 汇率/费率：保留 4 位小数
- 百分比：数据库存储为小数（如 0.15 而非 15），显示时乘 100
- 舍入模式：`ROUND_HALF_UP`（四舍五入）

**时间口径规则**：
- 服务端统一使用 UTC（`timezone.utc`）
- 前端根据用户时区显示
- "月"默认为自然月，财务月需显式标注
- 跨月数据归属以**订单/事件发生时间**为准，非入库时间

**关键域 acceptance checklist**：

| 域 | 验收项 |
|----|--------|
| 收入（MyIncome） | 月收入 = 基本工资 + 提成 + 奖金 - 扣款；各项加总 = 总额 |
| 绩效（Performance） | 评分公式结果与手动计算一致；排名无并列遗漏 |
| 目标（Target） | 月目标 = 日拆解之和；完成率 = 实际/目标 * 100 |
| 财务报表 | 收入 - 支出 = 利润；本月 + 上月 = 累计 |

## Risks / Trade-offs

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 规则过多导致 Agent/团队负担加重 | 开发效率下降 | 分三批落地（P0 立刻 / P1 第二批 / P2 第三批） |
| 压测环境与生产差异 | 压测结果不可靠 | staging 环境尽量对齐生产配置 |
| 恢复演练需要停服窗口 | 影响可用性 | 使用独立恢复环境，不占用生产 |
| 前端模板可能不适合所有页面类型 | 强制使用导致代码扭曲 | 模板设计为可扩展骨架，非刚性约束 |
